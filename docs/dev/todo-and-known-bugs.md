## Done - BUG-H (2026-06-07 vessels-disappearing): a stock Revert silently deleted the player's REAL landed/orbiting craft from unrelated missions

**Symptom (player-reported):** after a normal forward-play flight, hitting stock Revert (Esc -> Revert to Launch / Prelaunch, NOT a Parsek rewind) made several of the player's real, separately launched vessels vanish from the save. Evidence: `logs/2026-06-07_2312_vessels-disappearing/KSP.log`; the 21:49:46 Revert-to-Prelaunch (reverting the fresh mission `L4-B2-U1-S8`) logged `StripOrphanedSpawnedVessels: removed 7 vessel(s)` (`R3-B2-U1-S7` LANDED, `R2-B2-S5` ORBITING, `R.2-S.3`, `R.1-S.3`, `R.1-S.2` x2, `R.1-S.1` SPLASHED) - all real craft from unrelated prior missions. Pre-strip `saves/c1/persistent.sfs` had 8 vessels; the live save dropped to 3.

**Root cause (`persistentId` is craft-baked, NOT launch-unique - see CLAUDE.md):** on a stock Revert, `RevertDetector` arms `RevertKind`, `ParsekScenario.OnLoad` collects the `VesselName` of EVERY committed recording with `SpawnedVesselPersistentId != 0` into a GLOBAL name set, and `StripOrphanedSpawnedVessels` removed any flightState protoVessel whose `vesselName` was in that set (matcher was NAME ONLY, skipping only PRELAUNCH + ghost-map). ProtoVessels carry the craft-baked name/pid, which KSP reuses on every launch of a design, so a real, separately launched craft was indistinguishable from a recording's spawned vessel and got deleted on ANY revert that loaded a quicksave containing it. Cross-checking the 7 by launch Guid: 6 had `SpawnedVesselPersistentId` == live pid but a CONCLUSIVE Guid MISMATCH (a different launch wrongly deleted); the 7th (`R3-B2-U1-S7`, pid 364626662) was a real vessel Parsek had legitimately ADOPTED (`SpawnedVesselPersistentId == VesselPersistentId`, Guid match) but from an unrelated mission. The adoption path itself (`VesselSpawner.MaterializedSourceVesselExists` -> `VesselLaunchIdentity.LiveVesselIsRecordedLaunch`) was already Guid-gated; the STRIP threw that gating away by keying on name, and `ReconcileSpawnStateAfterStrip` keyed on bare pid.

**Fix (branch `bug-h-revert-vessel-strip`):** two corrections plus a centralized guid-aware deletion predicate so no strip/cleanup path can match a relaunch of the same craft. (1) **Launch-Guid gate.** New `VesselLaunchIdentity.LiveVesselIsRecordedSpawn` (genuine Parsek spawns use a KSP-unique spawn pid -> pid match conclusive; adoption stamps reuse the craft-baked pid -> require the launch Guid to not conclusively differ) + `CandidateMatchesRecording` / `FindMatchingRecording`. Every deletion routes through the centralized pure predicate `ParsekScenario.ShouldStripVesselForRecordings`, which resolves the candidate's pid + `Vessel.id`/`ProtoVessel.vesselID` Guid (the stale "ProtoVessel doesn't expose pid" comment was wrong and is removed) and only strips on a same-launch match. A conclusive Guid mismatch is NEVER stripped. (2) **Scope to the reverted flight.** The stock revert events fire with the state being left, not the target, so `RevertDetector` now captures the revert-target vessel pids from `FlightDriver.PreLaunchState` (Prelaunch) / `PostInitState` (Launch) `GameBackup.Config` (`ExtractFlightStateVesselPids`: GAME -> FLIGHTSTATE -> VESSEL[] persistentId) as a pre-existing whitelist; the revert strip removes only vessels NOT in that whitelist (i.e. they appeared during the reverted flight) and FAILS CLOSED (strips nothing) when the whitelist is unavailable. This protects the legitimately-adopted same-launch vessel (the 7th) that a pure Guid gate alone would still delete. `StripOrphanedSpawnedVessels` was rewritten to take the committed recordings + flags + whitelist; both callers updated (revert: matchSource=false, scope=true; rewind: matchSource=true, scope=false to preserve the rewind contract - genuinely future-timeline vessels are still removed via the launch-identity match plus the existing `StripFuturePrelaunchVessels` pid-whitelist pass). `ReconcileSpawnStateAfterStrip` gained a guid-aware overload (a surviving same-pid DIFFERENT-launch vessel no longer keeps a recording's spawn state alive); the revert reconcile routes through it. `ParsekFlight.CleanupOrphanedSpawnedVessels` (OnFlightReady belt-and-suspenders for FLIGHT->FLIGHT launch revert) now uses the same predicate + scope instead of name/pid recovery. `ShouldStripFuturePrelaunch` audited: pid-whitelist-based (keep-if-present), not in the defect class, behaviour unchanged. Tests: `VesselLaunchIdentityTests` (spawn/adoption/source matchers), `RevertVesselStripTests` (pid-collision-different-Guid not stripped; same-launch adoption protected by scope; genuine during-flight spawn stripped; fail-closed; rewind matchSource contract; guid-aware reconcile; `ExtractFlightStateVesselPids` parser), and in-game `RuntimeTests.RevertStrip_LiveGuidGate_ProtectsDifferentLaunch`. Note: recovering the already-deleted vessels in save `c1` is a separate manual repair, not part of this code fix.

## Done - BUG-F (2026-06-07 career-load regression): cold-loading an established career wiped the stock economy (funds to seed, science to 0)

**Symptom (player-reported):** loading the `c1` career save (build 0.10.1, all recent fixes merged) collapsed funds to 20095 (= seed 25000 - 4905 spent) and science to 0, with NO Parsek feature in use. The live save was overwritten on first cold-load. New-session log: `logs/2026-06-07_1934_career-load-regression/KSP.log`; pre-wipe backup `Kerbal Space Program/saves/c1_backup_pre_fix_20260607` (funds 72906, sci 1.0).

**Root cause (latent, not a 0.10.0->0.10.1 regression):** the load-time ledger recalc applies a "current-UT cutoff" that filters committed actions to the live game clock (built for post-rewind / future-timeline saves so future rewards are not paid out early). During a cold `ScenarioModule.OnLoad` the universe clock is not yet initialized, so `Planetarium.GetUniversalTime()` returns 0; the load passes `maxUT=0`. `LedgerOrchestrator.OnKspLoad` then took the cutoff path because `HasActionsAfterUT(0)` is true (every committed action is after UT 0), filtering 141 of 144 actions and patching funds/science down to the seed (`Current-UT ledger recalculation: reason=ksp-load cutoffUT=0` -> `PatchFunds: 72906 -> 20095`, `PatchScience: 1.0 -> 0.0`). The `DeferredSeedAndRecalculate` coroutine that was supposed to re-seed correct values fired with the same UT=0 (it waited for resource singleton VALUES, which were already loaded, not for the clock) and re-confirmed the wipe (`reason=deferred-seed cutoffUT=0`). The 0.10.0 session (`logs/2026-06-07_1638_career-playtest/`) only looked fine because that load had an empty ledger (`before=0, validRecordingIds=0` -> `HasActionsAfterUT(0)=false` -> safe full replay); both builds ran identical code with `maxUT=0, preserveFutureTimelineActions=True`. The in-session FLIGHT->FLIGHT load path was already guarded (`ShouldUseCurrentUtCutoffForPostRewindFlightLoad` checks `planetariumReady`); only the cold-load `OnKspLoad` + `DeferredSeed` paths were unguarded.

**Fix (branch `bug-f-coldload-wipe`, PR #1092):** added `LedgerOrchestrator.IsCurrentUtReadyForCutoff(double ut) => ut > 0.0` and gate every load-time cutoff/prune decision on it. (1) `OnKspLoad` (`LedgerOrchestrator.cs`) applies the current-UT cutoff only when `useCurrentUtCutoffForFutureActions && IsCurrentUtReadyForCutoff(maxUT) && HasActionsAfterUT(maxUT)`, otherwise falls back to the safe full replay (`RecalculateAndPatch`, cutoffUT=null) that replays every committed action; logs an `OnKspLoad recalc decision` line. (2) Both `Ledger.Reconcile` calls now pass `preserveFutureTimelineActions: useCurrentUtCutoffForFutureActions || !currentUtReady`, so a not-ready clock preserves all committed rows regardless of scene. This closes a sibling catastrophe found in review: a cold-load into a NON-cutoff scene (TRACKSTATION / EDITOR, where `useCurrentUtCutoffForFutureActions` is false) at maxUT=0 would otherwise let Reconcile PERMANENTLY PRUNE the whole career's spend history (not just mis-patch it). (3) The shared `RecalculateAndPatchForCurrentTimelineIfFutureActions` helper got the same guard (in-session callers pass a valid UT via `GetCurrentTimelineUTForLedgerRecalc`, so only the not-ready path changes - belt-and-suspenders against any future caller passing UT<=0). (4) `DeferredSeedAndRecalculate` (`ParsekScenario.cs`) now adds a bounded Phase-3 wait for the universe clock to report a positive UT before deciding, then routes through the guarded helper; a timed-out wait falls back to full replay. The per-frame clock poll reads through `TryReadReadyUniverseTime`, which wraps `Planetarium.GetUniversalTime()` in a try/catch (it can throw during very early load - same precedent as `Update()`), so a throw cannot kill the coroutine. This preserves the legitimate post-rewind case: a valid positive UT with committed actions still ahead of it keeps the cutoff (future rewards stay filtered). Tests: `IsCurrentUtReadyForCutoff_*` predicate; `OnKspLoad_ColdLoadWithUnreadyClock_ReplaysEntireCareer_NoWipe` (the regression test: maxUT=0 -> full replay, actionsAfterCutoff=N, no `reason=ksp-load` cutoff); `OnKspLoad_ColdLoadNonCutoffScene_PreservesCareerRows_NoPrune` (the prune sibling); `OnKspLoad_ReadyClockWithFutureActions_StillAppliesCutoff` (post-rewind case still cuts off); `OnKspLoad_ReadyClockPastAllActions_ReplaysFullLedger` (guard does not over-trigger); `RewindUtCutoffTests.CurrentTimelineIfFutureActions_NotReadyClock_ReplaysFullLedger` (shared helper guard); plus the existing `OnKspLoad_WithFutureLedgerActionsAndCurrentUtCutoff_RecalculatesAtLoadUt`.

**Follow-up (NOT in this fix, separate save/ledger repair):** after BUG-F lands, the `c1` save recalcs back only to its pre-wipe state (sci ~1.0, funds ~72906); fully restoring the lost Mun science (~124.8 - a BUG-A symptom) is a separate save repair.

## Done 2026-06-07 - Recalc-patch drawdown guard (defense-in-depth, BUG-A follow-up)

**Source:** The anticipated BUG-A follow-up (see the BUG-A entry below: "a defense-in-depth guard that refuses a large negative drawdown when no rewind/re-fly is active is a sensible follow-up but was not needed to fix the root cause"). Plan: `docs/dev/plans/recalc-patch-drawdown-guard.md`.

**What it does:** When a ledger recalc would patch live funds / science / reputation DOWN past the per-resource rounding epsilon (funds 0.01, science 0.001, rep 0.01) AND no time-travel context is active, the patch is CLAMPED so the player's value is preserved (`max(patchTarget, currentLive)`), a single session-scoped native `ParsekLog.ScreenMessage` ("Kept your earned funds/science/reputation") is posted per resource, and a WARN with the full numbers fires on every guarded recalc. Always on, no setting.

**Predicate (Blocker 2 / plan §3.3):** The guard keys on the NON-RESERVED running balance (`FundsModule.GetRunningBalance` / `ScienceModule.GetRunningScience` / `ReputationModule.GetRunningRep`), NOT the reservation-aware patch target (`GetAvailableFunds` / `GetAvailableScience`). A reservation legitimately lowers `available` below `running` on cutoff walks (committed future spends), so keying on the target would defeat overspend prevention; a missing-earning leak lowers `running` itself below live. Reputation has no reservation, so its running balance IS its target. Per-subject science (`PatchPerSubjectScience`) stays unclamped (a documented limitation called out in the pool clamp WARN, MINOR 5).

**Authoritative-reduction bypass (five signals, plan §3.2):** the clamp is suppressed when any of `RewindContext.IsRewinding`, `ParsekScenario.Instance.ActiveReFlySessionMarker != null`, `ParsekScenario.Instance.ActiveMergeJournal != null`, the tombstone path (`RecalculateAndPatchAfterTombstones` -> `tombstonePath: true`), or the NEW `RewindContext.RewindResourceAdjustmentInProgress` is set. The fifth signal (Blocker 1) is the ONLY one true at the deferred plain-Rewind-to-Launch authoritative recalc (`ApplyRewindResourceAdjustment` -> `RecalculateAndPatch(adjustedUT)`): it is set synchronously in `HandleRewindOnLoad` before `StartCoroutine`, cleared in the coroutine's try/finally around that patch (authoritative), and cleared as a race-free fail-safe in the next scene's `ParsekScenario.OnAwake` (NOT a blanket scene-switch clear, which would race the coroutine's ~2s singleton wait).

**Plumbing:** `RecalculateAndPatchCore` computes `authoritativeReduction` from the five signals and threads it (plus a new `tombstonePath` Core param) through `ApplyRecalculatedStateToKsp` -> `KspStatePatcher.PatchAll` -> `PatchScience` / `PatchFunds` / `PatchReputation`, parallel to the existing `suppressSuspiciousDrawdownWarnings`. The legacy log-only `IsSuspiciousDrawdown` 10% WARN is untouched.

**Files:** `Source/Parsek/GameActions/KspStatePatcher.cs` (`IsGuardableDrawdown`, `ApplyDrawdownGuard`, `EmitDrawdownGuardClamp`, session-scoped toast latches, three patch methods), `Source/Parsek/GameActions/LedgerOrchestrator.cs` (`IsAuthoritativeReduction`, Core/Apply threading), `Source/Parsek/RewindContext.cs` (signal 5 + Begin/End + ResetForTesting), `Source/Parsek/ParsekScenario.cs` (HandleRewindOnLoad arm, coroutine finally, OnAwake fail-safe, save-boundary latch reset), `Source/Parsek/InGameTests/RuntimeTests.cs` (live clamp + signal-bypass test), `Source/Parsek.Tests/DrawdownGuardTests.cs`.

**Status:** Fixed for v0.10.1. Full xUnit suite green (14559, +25). In-game validation pending: a clean career playtest must produce ZERO guarded clamps; a deliberately-leaked / forced-too-low ledger must clamp + preserve live + toast on funds/science/rep; a real rewind / re-fly / tombstone reduction must still apply (authorized). Run `RuntimeTests.DrawdownGuard_ClampsLeakAndAuthorizesSignal` (Ctrl+Shift+T, SPACECENTER) to verify the live patch path.

## Done 2026-06-07 - BUG-G: tech-purchase affordability gate contradicted the drawdown guard AND blocked destructively (double science loss)

**Source:** 2026-06-07 career playtest, `logs/2026-06-07_2110_tech-purchase-bug/KSP.log` (~lines 13027-13045). Pure-stock play, no Parsek feature used. The player had live science 124.8 but Parsek blocked a 45-cost tech unlock as "insufficient science" AND the science was deducted anyway, twice (`ScienceChanged -45.0 -> 79.8`, block, `ScienceChanged -45.0 -> 34.8`, block; ~90 science lost, no node unlocked). This is the missing OTHER HALF of the drawdown guard above: the guard preserves live above a leaked ledger running balance, but the affordability gate read the leaked ledger value.

**Two facets.**

**Bug 1 (gate vs guard inconsistency):** `LedgerOrchestrator.CanAffordScienceSpending` / `CanAffordFundsSpending` computed `available` from the ledger (`GetAvailableScience` / `GetAvailableFunds`). With a missing-earning leak (running below live, no time-travel context, no reservation: `futureActions=0` in the log) `available` was 1.0 while the drawdown guard had preserved live at 124.8, so the gate blocked a purchase the player could afford. Fix: new pure helper `LedgerOrchestrator.ComputeEffectiveAffordable(available, running, live, authoritativeReduction)` returns `authoritativeReduction ? available : available + max(0, live - running)`, adding back exactly the guard-preserved leak gap while a GENUINE reservation (`available < running`) still reduces spendable below the leak-corrected base (so it is NOT a plain `max(available, live)`, which would defeat overspend prevention). The live wrappers apply it only when the KSP singleton exists (real game); xUnit keeps the pure-ledger contract so the existing reservation/cutoff `CanAfford*` tests stay authoritative. `IsAuthoritativeReductionForAffordability` reads the same five drawdown-guard signals (tombstonePath=false).

**Bug 2 (block was destructive):** stock `RDTech.ResearchTech` deducts science (`host.AddScience(-scienceCost, RnDTechResearch)`) and ONLY THEN calls `UnlockTech`, so the prior `UnlockTech`-prefix block fired AFTER the deduction and returned false without refunding (silent loss). The facility equivalent is the same: `SpaceCenterBuilding.UpgradeFacility` deducts funds then calls `SetLevel`. Fix: gate BEFORE the deduction. New `TechResearchSpendPatch` (prefix on `RDTech.ResearchTech`) and `FacilityUpgradeSpendPatch` (prefix on `SpaceCenterBuilding.UpgradeFacility(bool)`) run the block pre-deduction; returning false skips the original entirely (no deduction, node/level unchanged, `RDController.ActionButtonClick` discards the result and `RefreshUI` re-reads the unchanged state). The shared decision lives in `TechResearchPatch.TryBlockTechResearch(tech, includeAffordability)` / `FacilityUpgradePatch.TryBlockFacilityUpgrade(facility)`. The post-deduction `UnlockTech` / `SetLevel` prefixes are kept as non-destructive backstops for direct callers and now block COMMITTED items only (the destructive affordability re-check was removed from `UnlockTech`).

**Files:** `Source/Parsek/GameActions/LedgerOrchestrator.cs` (`ComputeEffectiveAffordable`, `IsAuthoritativeReductionForAffordability`, both `CanAfford*` wrappers), `Source/Parsek/Patches/TechResearchPatch.cs` (`TryBlockTechResearch`, UnlockTech committed-only backstop), `Source/Parsek/Patches/TechResearchSpendPatch.cs` (NEW), `Source/Parsek/Patches/FacilityUpgradePatch.cs` (`TryBlockFacilityUpgrade`), `Source/Parsek/Patches/FacilityUpgradeSpendPatch.cs` (NEW), `Source/Parsek.Tests/SpendingGateConsistencyTests.cs` (NEW), `Source/Parsek/InGameTests/RuntimeTests.cs` (`SpendingGate_RespectsLiveAndBlocksNonDestructively`).

**Status:** Fixed for v0.10.1. Full xUnit suite green (14596 pass; only the environmental `InjectAllRecordings` flake fails when KSP holds the install). In-game validation pending: run `RuntimeTests.SpendingGate_RespectsLiveAndBlocksNonDestructively` (Ctrl+Shift+T, SPACECENTER); also confirm in a real R&D scene that a leaked-ledger career can still buy affordable nodes and that any block leaves science/funds unchanged. Note: the affected `c1` save was repaired by hand (ledger seeds bumped so running matches live); this fix prevents recurrence + stops the destructive block, it does not heal saves.

## Done - BUG-B (2026-06-07 career playtest): Parsek auto-spawned ghosts, map icons, and duplicate recorded vessels during a normal playthrough (no rewind/playback used)

**Symptom (player-reported):** a Parsek ghost icon was visible in map view, and Parsek spawned then re-recovered real vessels named after recordings, during a pure-stock career where NO Parsek feature (rewind, re-fly, loop, preview) was used. Bug list + 76 MB log: `logs/2026-06-07_1638_career-playtest/BUGS.md` (BUG-B).

**Root cause:** the ghost-playback engine (`ParsekFlight.UpdateTimelinePlaybackViaEngine`, every frame) and the Space Center path (`ParsekKSC`) evaluate every committed recording purely by comparing the live game UT to the recording's recorded window. Per the README "How It Works", a recording replays (and spawns its terminal vessel) only AFTER the player rewinds to before it, so its window lies in the player's forward path again. There was no gate distinguishing that from normal forward play, where KSP's UT only advances, so every committed recording's window is already historical the moment it is committed. The engine therefore treated "live UT > recorded end" as "playback finished" and fired `PlaybackCompleted` -> terminal-vessel spawn for every spawnable recording (the log shows `needsSpawn=True` with `ghostWasActive=False`, an impossible combination for a genuine replay), while orbital recordings whose recorded/extrapolated END runs ahead of the live UT (e.g. `R2-B2-S7` recUT=[203566.8,210582.4] at liveUT ~203585) rendered a live ghost + map icon of the player's still-orbiting vessel. The duplicates were re-recovered each scene by `CleanupOrphanedSpawnedVessels`, and the auto-spawned terminal-orbit vessels are the UPSTREAM trigger for BUG-C's terminal-orbit spawn-death / abandonment churn: BUG-C signatures 2-3 (`R2-B2-S5` abandoned 3x; fresh-rollout craft-baked-pid mis-identification) are fallout of this auto-spawn and are removed at the root once historical recordings stop auto-spawning, while BUG-C signature 1 (NaN debris -> stock `FlightIntegrator` throw) is pure stock KSP and unaffected. (The earlier "orphan-recovery refunds feed BUG-A's funds clawback" hypothesis was investigated separately and RULED OUT: those recoveries fired for unrelated `R.1` vessels and BUG-A's -47840 was dropped Mun-firsts milestone funds, not recovery credit.)

**Fix (branch `bug-b-ghost-autospawn`):** new `PlaybackScopeTracker` per-recording latch keyed by recordingId. A recording enters "replay scope" the first frame the live playhead is observed at or before its activation-start UT (the post-rewind case). The discriminant keys on the recording's START, which stays behind the playhead during forward play even when the orbital-extrapolated END runs ahead, so the orbital-icon case is caught too. A recording NOT in scope, whose window the playhead only ever moved forward through/past, is `HistoricalNotReplayed`: flight `ComputePlaybackFlags` sets `skipGhost` (new `GhostPlaybackSkipReason.HistoricalNotReplayed`, reusing the existing skip-render path so there is no ghost and no deferred map presence) and `needsSpawn=false` (no policy terminal spawn); `ParsekKSC` suppresses the non-loop render and the playback-disabled past-end spawn. Looping / chain-loop / mission-unit members (explicit live-replay opt-in) and the active re-fly session are exempt and keep rendering. Rewind-to-launch naturally re-arms scope (the playhead drops below the target's start); a rewind to a LATER launch correctly leaves earlier recordings historical (a single global "replay active" flag would get this wrong, which is why the latch is per-recording). The gate is wired in all three ghost-rendering scenes: flight (`ComputePlaybackFlags` -> skipGhost + needsSpawn=false), Space Center (`ParsekKSC` non-loop render + disabled past-end spawn), and Tracking Station (icon path: `GhostMapPresence.ResolveTrackingStationGhostSourceCore`, gated on `batch != null` so only the live create / lifecycle paths are affected and the pure `ResolveTrackingStationGhostSource` predicate keeps its old behaviour; spawn path: the per-frame `TryRunTrackingStationSpawnHandoffs` driver also skips the historical terminal-vessel spawn, exempting the adoption case where a matching live vessel is still present, while the explicit "Warp to Spawn" action `TryRunTrackingStationSpawnHandoffForIndex`/`ForRecordingId` stays ungated as a deliberate user action). Tests: `PlaybackScopeTrackerTests` (pure discriminant incl. orbital-extrapolation, rewind-to-later-launch, and tolerance-boundary cases) + three `FlightPlaybackExplainabilityTests` end-to-end cases (historical -> skip + no spawn, rewound-into-scope -> renders + spawns, looping -> exempt).

Known limitations / residual edges (all OUTSIDE the reported no-Parsek-feature case, which is fixed unconditionally): (1) the in-memory latch is keyed by recordingId and cleared only on main-menu transition, so it survives a quickload (F9). A replay paused mid-flight and quickloaded reverts its recordings to historical (benign: a replay stops); conversely a quickload to a forward-play state AFTER a rewind keeps the stale in-scope mark, so the rewound recordings can render again until the next main-menu cycle. (2) On the single frame a Mission first begins looping, `ComputePlaybackFlags` reads the previous frame's `cachedLoopUnits` (rebuilt after the flag pass), so a mission-unit member can flicker its ghost for one frame before re-rendering. (3) Flight exempts `loopingLike` from the historical flag while the KSC playback-disabled past-end branch does not, so a `LoopPlayback=true` + `PlaybackEnabled=false` recording is gated at the Space Center but not in flight; the suppress-during-forward-play behaviour is the more correct of the two.

## BUG-C (2026-06-07 career playtest) - `R2-B2` tree instability + NaN debris -> stock exceptions

Source: `logs/2026-06-07_1638_career-playtest/` (KSP.log, `BUGS.md` BUG-C section). Build `Parsek V0.10.0` @ `07dea8fac`. The player used NO Parsek features this session (no rewind / re-fly / loop / playback); Parsek was only background-recording. Three log signatures, separable root causes. BUG-C is largely fallout of BUG-A (ledger recalc) and BUG-B (passive ghost / vessel auto-spawn), which are tracked separately.

### 1. NaN debris -> stock `FlightIntegrator.UpdateOcclusionSolar` throw - STOCK KSP, not Parsek data (no fix)

Two `ArgumentOutOfRangeException` throws in stock `FlightIntegrator.UpdateOcclusionSolar` (KSP.log lines 205423 @15:46:42, 218683 @15:50:46), each immediately around `R2-B2M-S6 Debris had a NaN Orbit and was removed` + an on-rails Kerbin->Sun SOI transition.

Origin is **pure stock physics**, confirmed:
- `R2-B2M-S6 Debris` (pids 1333358833 / 2800168062) is the player's **real staging debris**, created at 15:42:22 by a real decouple (`Decouple created vessel during recording ... rootPart=radialDecoupler`), with real drag cubes, terrain collision (`crashed through terrain on Kerbin`), and explosions. It is NOT a Parsek ghost/spawn: line 204856 `CleanupOrphanedSpawnedVessels: no match for 'R2-B2M-S6 Debris'` is Parsek explicitly disclaiming ownership.
- Parsek background-recorded the debris and then **finalized + deleted** those recordings as non-persistable (`canPersist=False`, `DeleteRecordingFiles`) at 15:42:39, ~4 minutes before the NaN. No Parsek recording carried or authored the NaN orbit.
- The throw is the well-known stock pattern: debris clips through terrain, is packed on rails with a degenerate velocity, the resulting hyperbolic orbit escapes Kerbin->Sun, and `UpdateOcclusionSolar` indexes a body list off a NaN-derived value and throws before stock's own NaN-orbit removal runs.

Decision: do **not** Harmony-patch stock `FlightIntegrator` to swallow this. It is not Parsek data, it reproduces without Parsek, and guarding a stock NaN path from a mod is high-risk for little gain (the proper home for a stock-bug shim is KSPCommunityFixes). Filed as known-stock, no Parsek code change.

### 2. Terminal-orbit ghost "permanently abandoned" 3x - BUG-B fallout + a real durability gap (FIXED here)

## ~~OPEN~~ RESOLVED 2026-06-06 (off-orbit fix true for ALL members; coverage gap closed; cutover COMPLETE at S4 - legacy ownership deleted + `mapRenderDirectorDrive` gate dropped; 8f icon-floor deletion REASSESSED + the floor KEPT as a permanent no-conic fallback) - v0.10.0 Looped ghost map icon rotated ~96.5 deg off its orbit lines (root cause CONFIRMED 2026-06-02; Phase-8a epoch-bake fix, now the single modular render path)

- Reported in playtest: on a looped re-aimed mission ("Kerbal X", rec 30), the map icon glided the parking orbit and the orbit-raise polyline correctly, then when the circular loiter orbit appeared the icon jumped ~90 deg to the wrong point on it instead of continuing from where the ascent ended. - **Diagnosis (instrumented, not guessed):** added `GhostIconTruth` (mna + inc/LAN/argPe + body-relative `orbit.pos` + body-fixed lat/lon), a full-rate `GhostIconJump` step detector, and a `GhostIconDiverge` detector comparing the orbit-line longitude (`orbitLon`, computed fresh from `orbit.pos` this frame) against the vessel-transform longitude (`transformLon`, from `GetWorldPos3D`). Findings, in order: (1) `orbit.pos` inertial longitude is perfectly smooth across the seam (21.88 -> 22.66 -> ...), so the orbit math and the effUT/`DecompressSpanUT` clock are NOT the bug, and the compressor stays innocent; (2) the `GetWorldPos3D` lat/lon "jump" was a stale-within-frame read (red herring); (3) the real divergence: at the loiter takeover `orbitLon` = 161.78 (orbit line) vs `transformLon` = 64.94 (raise end), `lonDiv` = 96.84 deg, at a loop shift of 845,044,936 s.
- **Root cause:** a looped recording replays its orbits at the RAW recorded epoch under a live-frame shift. The orbit keeps its recorded INERTIAL orientation (LAN), but the reference body keeps rotating, so over the shift Kerbin turns 845044936 / 21549.425 = 39214.27 rev -> 0.268 rev = 96.62 deg, and the inertial orbit's BODY-FIXED projection at the live clock is rotated 96.62 deg from the recording. With a whole-rotation shift the body-fixed raise and the inertial orbits would coincide and there would be no teleport; the interplanetary re-aim breaks that.
- **Why this is a single-frame seam, not multiple bugs:** in the recording the raise injection and the loiter start are the SAME point; the 96.6 deg gap is purely a rendering-frame seam between the surface-locked raise and the inertial orbits. ### CONFIRMED root cause (2026-06-02, `angleIconVsOrbitEff` metric + KSP decompile) and the implemented fix
- **What the metric proved:** with `mapRenderTracing` on, the gated `icon-off-orbit` anomaly logged (steady state, s15 "Duna One", pid 2333335827) `angleIconVsOrbitEff=96.50 angleEffVsLive=96.73 lonIcon=31.11 lonOrbitEff=-65.39 lonOrbitLive=31.34 iconR=731903 orbitEffR=731903`. Three facts: (1) `iconR == orbitEffR` -> pure rotation, same orbital altitude (NOT a body-fixed surface point); (2) `lonIcon ~= lonOrbitLive`, NOT `lonOrbitEff` -> the icon sits at the orbit evaluated at the LIVE clock, while the line is drawn at effUT; (3) `angleIconVsOrbitEff ~= angleEffVsLive` -> the whole gap IS the orbital phase advance over the loop shift. So the bug is a PHASE (clock) mismatch, not orbit elements and not a body-fixed projection.
- **Why the effUT drive never reached the icon:** decompiling `Vessel.GetWorldPos3D` -> returns `CoMD`; `VesselPrecalculate` rebuilds a packed non-landed vessel's `CoMD = mainBody.position + orbitDriver.pos` EVERY FixedUpdate, and `orbitDriver.pos` is re-propagated at the LIVE Planetarium clock. So `GhostOrbitIconDrivePatch`'s `UpdateFromUT(effUT)` + `SetPosition` is overwritten by KSP's live re-propagation; the icon always ends up at `orbit(liveUT)` regardless. The raw-epoch + per-frame-effUT-drive architecture was built on the premise that the drive controls the icon - it does not.
- **The fix (Phase-8a director-drive, gated `mapRenderDirectorDrive`, default on as of 2026-06-05):** bake the loop shift into the orbit EPOCH (`StockConicTreatment.SeedAndDriveLive`: `SetOrbit(..., epoch + shift, ...)`) and propagate at the LIVE clock. Then `M(live) = MAE + n*(live - epoch - shift) = MAE + n*(effUT - epoch)` = the recorded mean anomaly at effUT, so the live re-propagation (whatever rebuilds `orbitDriver.pos`) lands on the recorded phase. The icon rides the same orbit the line is drawn from. This is the IDENTICAL phase the legacy effUT-drive intended (raw epoch propagated at `live - shift` gives the same mean anomaly), so it is not a new orbit and moves PHASE only - LAN/inc/argPe untouched, so it is NOT the reverted per-element LAN-rotation dead-end. ## Closed (not a bug) - Logistics send-once re-arm delivers at the NEXT dock crossing, up to a full span away

- Playtest observation (save "orbital supply route DELIVERY test", 2026-06-06): a KSC -> Depot supply route (recorded span ~16058s, recorded dock UT 17478, span end ~17594) delivered correctly on its first cycle (`Route ... Active->Paused reason=delivered-then-paused completedCycles=1`), the player re-armed a single cycle (`Route Paused->Active reason=send-once-arm`), and no second delivery fired in the ~59s before the scene changed. Every dispatch tick logged cleanly (`dispatched=0 ... errored=0 skipped=1`): no error, no stuck/errored state.
- **Verdict: correct by design, not a bug.** A v0 supply route is a looped Mission segment that renders `[root launch .. dock]` and **delivery fires when the loop clock reaches the recorded dock PHASE, not at cycle start** (design section 0.4 "Delivery fires when that loop clock crosses the recorded dock UT each cycle" and section 0.9 DEL-2 "Delivery fires when the loop clock reaches the recorded dock phase, not at cycle start"). One delivery == one full ghost run reaching its dock.
- **Why dock UT sits at ~99% of the span (NOT an anchor / off-by-one bug):** the route SEGMENT is `[launch .. dock]` by construction. `RouteBuilder` computes `transitDuration = recordedDockUT - rootLaunchUT` ([RouteBuilder.cs:153](../../Source/Parsek/Logistics/RouteBuilder.cs)) and the dock UT "is the route segment END ... with the [launch..dock] segment it equals spanEnd" ([RouteBuilder.cs:387-389](../../Source/Parsek/Logistics/RouteBuilder.cs)). The small gap between dock (17478) and span end (17594) is the post-dock recording tail / boundary epsilon; the dock is still inside the span (`RouteLoopClock.IsDockUTInSpan` true), so a crossing can fire. Moving the dock earlier in the span would deliver BEFORE the ghost visually docks, breaking the core "deliver at the dock" contract; the dock is genuinely the end of an outbound launch->dock run.
- **Why the second cycle did not fire in the window (the mechanism, confirmed):** after delivery #1 the dock-phase cycle was cycle 0, so `LastObservedLoopCycleIndex = 0`. The loop clock free-runs on absolute UT (it is NOT advanced by the orchestrator while paused, and re-arm does not reset it). At re-arm the clock had already rolled past cycle 0's dock into cycle 1 and was early in the run (`loopUT` ~3283, well below the dock at 17478), so `ComputeDockCycleIndex = cycleIndex - 1 = 0` ([RouteLoopClock.cs:154-157](../../Source/Parsek/Logistics/RouteLoopClock.cs)) and `IsDockCrossing` returns `dockCycleIndex (0) > LastObservedLoopCycleIndex (0)` == false ([RouteLoopClock.cs:190-202](../../Source/Parsek/Logistics/RouteLoopClock.cs)). The next delivery requires the clock to climb back around to the dock of cycle 1 (`loopUT` 3283 -> 17478, ~14000s of ghost-time), which the player's ~59s of warp did not cover.
- **The gate is exactly right (no extra forced wait):** the `>` comparison (not `>=`) is necessary so cycle 0's already-delivered dock is not re-delivered. The gate fires at the next UN-delivered dock crossing, firing exactly once and snapping forward on a warp jump ([RouteOrchestrator.cs ProcessLoopRoute](../../Source/Parsek/Logistics/RouteOrchestrator.cs), `crossing` branch). Send-once correctly does NOT reset `LastObservedLoopCycleIndex` (unlike `TryActivate`): resetting it to -1 would make a re-arm fire IMMEDIATELY while the ghost is mid-flight (`dockCycleIndex - 1 = 0 > -1`), delivering with no ghost at the dock, which is the wrong contract.
- **Intended send-once cadence (so this is not re-investigated):** re-arming send-once schedules ONE delivery at the loop clock's next un-delivered dock crossing. Because the dock is the end of the launch->dock segment and the clock free-runs, the wall-clock-UT wait is anywhere from ~0 (if a dock passed during the pause, in which case the next tick fires and snaps forward) up to one full span (`DispatchInterval = N x span`, default N=1). Re-arming right after a delivery lands near the worst case: you must warp the ghost through nearly a full outbound run for it to dock again. The Logistics window's "Next delivery" countdown (`RouteOrchestrator.TryComputeSecondsToNextDockCrossing`) surfaces this expected wait. This is the same near-full-span warp the FIRST send-once needs to reach the dock; it is fundamental to delivery == the ghost docking, not a regression.
- **Related observation (same playtest) - "the send-once ghost exploded / disappeared seconds after launch" is faithful ascent staging plus the loop overlap/zone behaviour under time warp, NOT the main vehicle dying.** Confirmed from the AUTHORITATIVE logistics-build log (`logs/2026-06-06_1408_logistics-supply-route/KSP.log`, `Parsek V0.10.0`, branch `logistics-v0-implementation` commit `f0855865` - this is the original 13:51-13:59 send-once session, captured by `collect-logs.py` BEFORE it was overwritten; an earlier read mistakenly concluded it was gone). ## Open - Logistics route window dock-side baseline is post-couple

- The `RouteConnectionWindow.DockTransportResources` / `DockEndpointResources` manifests captured in `ParsekFlight.CreateMergeBranch` come from the merged vessel's `mergedSnapshot` (built from the live merged vessel taken inside `onPartCouple`). By the time the snapshot is taken the parts are physically joined, so any same-frame stock crossfeed equalisation between transport and endpoint tanks of the same resource lands on the dock-side baseline, not the undock-side delta. The strict `RouteAnalysisEngine.HasMixedPickupDelivery` gate then sees `transportGain > 0` on a resource the player never intended to move and rejects the run as `MixedPickupDelivery`.

The route-analyzer rejection logs at `Info` ("RouteAnalysis rejected: mixed pickup/delivery") and now names the tripping resource (`pickup resource=<name> ...`) so the affected tank's resource is visible in `KSP.log` without raising the verbosity dial. (ElectricCharge and IntakeAir are filtered out and never trip this gate; see the EC/IntakeAir filter item above.)

  - One-line description (`Contract "Rescue Kerbal" deadline UT 240000 is before committed completion at UT 250000`).
  - `Go to ...` button linking to the relevant other window (Timeline scroll, Career State tab, etc.).
- Warnings are computed once per `OnTimelineDataChanged` fan-out (same cache-invalidation channel everything else uses).
- Starter rule set, each as a pure static helper in `WarningRules.cs`:
  - **ContractDeadlineMissed** — active contract's `DeadlineUT < terminal-UT of its committed completion recording`.
  - **FacilityLevelRequirement** — an action requires facility level N but the facility doesn't reach N until after that action's UT.
  - **StrategySlotOverflow** — projected active strategies > projected max slots (currently only warned in log, not UI).
  - **ContractSlotOverflow** — same for contracts.
  - **CrewDoubleBooking** — a stand-in appears in two chains at overlapping UT ranges.
  - **ResourceOverCommit** — already shown in Timeline budget footer, but also listed here for one-stop-shop.

**Why it matters:**

Action blocking catches paradoxes at the moment the player tries to violate them. Warnings catch *latent* contradictions that the ledger can detect but won't error on — the subtle ones where the ledger silently picks a resolution the player didn't intend (e.g. contract gets zeroed out because its deadline passed unexpectedly). Surfacing these early turns the mod's "structural paradox prevention" into a communicated design contract rather than a hidden invariant.

**Files to touch:**

- `Source/Parsek/UI/WarningsWindowUI.cs` — new scrollable list window.
- `Source/Parsek/WarningRules.cs` — new pure-static rule evaluators, one method per rule, each returning `List<Warning>` given `(ledger, recordings, modules)`. Heavy unit-test coverage.
- `Source/Parsek/ParsekUI.cs` — add the badge button + open toggle; integrate with `OnTimelineDataChanged` cache invalidation.
- `Source/Parsek.Tests/WarningRulesTests.cs` — one test per rule (happy + each flag condition).

**Out of scope for v1:**

- Auto-fix for any warning. Pure read-only surface.
- Severity levels / color-coding. All warnings are equal in v1; add severity in a follow-up if there are too many of one kind.
- Per-rule disable toggles. Playtesting can decide which rules feel noisy before we add knobs.

**Status:** TODO. Size: M. Complements the help popup (#426) — where help explains the system, warnings explain *your career's* specific issues. Together they turn the mod from "learn by experimenting" to "learn by seeing the model."
