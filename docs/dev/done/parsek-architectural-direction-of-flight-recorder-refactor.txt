Architectural direction for the flight recorder refactor

Parsek's ghost recording and playback system will eventually be extracted into a standalone mod — a general-purpose ghost loop playback engine that Parsek depends on as a hard dependency. This standalone mod will handle trajectory recording, ghost GameObject rendering, part event replay, rendering zones, and soft caps. It will also ship its own minimal recorder for single-vessel flights with staging, and a content pack system for loading pre-authored ghost loops (background KSC traffic, scenery, etc. — including non-vessel content like birds, which means the renderer can't be permanently coupled to KSP's part prefab system).

Parsek will become a consumer of this mod's API, adding its own layers on top: the recording tree (DAG), merge events (docking/boarding), chain walker, spawn policy (chain-tip spawning, intermediate suppression, first-loop-is-real), CommNet relay registration, tracking station and map view presence, game actions, timeline, and rewind. The standalone mod has no concept of any of these — it plays trajectories as visual ghosts and nothing more.

The extraction itself is not happening yet — it will happen after both the flight recorder redesign and the game actions redesign (plus their respective refactors) are complete. But this refactor is where we prepare the code so the eventual extraction is a clean move rather than a painful untangling.

Here's what that means concretely for this refactor:

1. The ghost renderer should only access trajectory and visual data through an interface, not by reading Recording objects directly. Recording carries tree linkage, resource deltas, spawn tracking, controller info, crew data — none of which the ghost renderer uses. Introduce an interface that exposes only what the renderer needs: trajectory frames, part events, time bounds, mesh snapshot, loop config, anchor info. Recording implements it. The renderer only sees the interface.

2. Ghost lifecycle management should separate into two layers: playback mechanics and policy. Playback mechanics means creating GameObjects, positioning them each frame, firing part events, managing zone transitions and soft caps. Policy means deciding whether a ghost should exist, whether it's a chain ghost or a loop, what happens when playback ends, whether spawn should be suppressed. These should be clearly separated — ideally two classes, at minimum two distinct regions of responsibility. The test: can you call the playback mechanics layer without any reference to RecordingTree, BranchPoint, chain walker state, or resource deltas? If yes, the boundary is right.

3. The "recording playback completed" signal should be event-based. When a ghost finishes its trajectory, downstream systems react (spawn decisions, resource processing). If this is currently a direct method call from ghost management into spawn/resource code, refactor it into a callback or event that listeners can hook. Same for ghost creation and ghost destruction — these should be hookable events, not hardwired calls. There's only one listener right now, but the architecture should support the pattern.

4. Visual SegmentEvents should be distinguishable from identity SegmentEvents. PART_DESTROYED, PART_ADDED, and TIME_JUMP affect ghost rendering — the ghost renderer needs these to update meshes and handle trajectory discontinuities. CONTROLLER_CHANGE, CONTROLLER_DISABLED, CONTROLLER_ENABLED, CREW_LOST, CREW_TRANSFER are identity and game-state concerns with no visual effect on the ghost. Make this distinction explicit — either separate lists, or tagging so the renderer can filter to only the visual events without knowing what a "controller" is.

5. The staging/split visual transition should be self-contained as a "segment group" concept. A staged rocket is not just one ghost — it's one ghost that becomes several playing in parallel when boosters separate. The ghost renderer needs to handle this transition seamlessly: at a split UT, the parent ghost mesh is destroyed, child ghost meshes appear at the correct positions, and each child plays its own trajectory independently until it expires (crash, despawn, TTL). On loop reset, all children are destroyed and playback restarts from the root. This entire visual transition should be expressible without walking the RecordingTree or understanding BranchPoints. The renderer should only need: a split UT, a part mapping (which parts went to which child), and trajectory data for each child segment. The tree/DAG is a higher-level concern — the renderer just plays a group of linked segments.

6. Trajectory data should not carry game-state fields. TrajectoryPoint currently includes funds, science, and reputation. The ghost renderer doesn't use these. Either strip them from the trajectory type and move them to a parallel game-state structure indexed by UT, or at minimum ensure the ghost playback path never reads them. Trajectory data should be purely positional and rotational. This also makes the eventual file format split cleaner — the standalone mod's trajectory format won't include these fields.

7. Game actions should have zero runtime dependency on ghost objects. The game actions system should never read ghost position, ghost state, or ghost existence directly. It should react to events ("recording playback completed at UT=X") not poll ghost state. Verify this is true in code and fix any coupling found. This is important both for the eventual extraction and for the upcoming game actions redesign — the game actions system needs to be a purely timeline-driven calculation engine that doesn't care whether ghosts are rendering or not.

8. Ghost mesh construction should be structured to eventually support non-part-based meshes. Right now, ghosts are built by cloning KSP part prefab meshes from a vessel snapshot. The standalone mod will eventually also need to render custom content (non-vessel meshes bundled in content packs). This doesn't mean building that system now — but the ghost construction code should be structured so that "build a ghost from a vessel snapshot" is one path through a more general "build a ghost from a mesh source" pattern, rather than the only hardcoded path. If the mesh builder has a clean entry point that takes a snapshot and produces a GameObject, that's enough — it can be generalized later without restructuring.

None of this changes external behavior. All existing functionality should work identically after the refactor. We're restructuring internals so the code is modular in shape — clean interfaces, separated concerns, event-driven coupling — even though it all still lives in one assembly for now.
