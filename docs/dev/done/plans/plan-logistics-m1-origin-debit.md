# Plan: Logistics M1 - non-KSC origin debit

**Branch:** `logistics-m1-origin-debit` (from origin/main tip, post-0.10.1)
**Spec:** `docs/parsek-logistics-supply-routes-design.md` section 19.4 "M1" (lines 1594-1603), constrained by 19.2.2 (provenance), 19.2.5 (gating semantics), and section 0.2 (Missions LOCKED).

## 1. Summary + scope

M1 makes docked-start non-KSC-origin routes dispatch. Six deliverables:

1. **Physical origin debit**: `LiveOriginDebitWriters` (mirror of `LiveDeliveryWriters`, removing from the origin vessel, loaded/unloaded write split) plus the `RouteCargoDebited` ledger row carrying the actual debited manifest, emitted on the same direct-write-plus-bookkeeping-row pattern the delivery side uses.
2. **`OriginHasCargo` un-stub**: resolve `route.Origin` to a live vessel, sum its deliverable stored resources, gate all-or-nothing against `route.CostManifest`.
3. **Origin endpoint descriptor capture**: populate situation + coordinates of the docked origin partner at recording start (recorder side, additive), so `RouteBuilder` can build a surface-typed origin endpoint that gets the same proximity rebuild fallback destinations have.
4. **Undocked-start workflow rejection**: new `RouteAnalysisStatus.UndockedStartOrigin` + player-language formatter text, surfaced through the existing near-miss pipeline.
5. **Dispatch priority**: persisted `Route.DispatchPriority` (additive codec), deterministic tick-order comparator (priority, then `NextDispatchUT`, then route id), minimal detail-panel stepper UI.
6. **Scenario-lifecycle test harness**: three-layer shape - xUnit codec round-trips (largely existing, extended), tightened ORDERED source-text gates, and new in-game lifecycle tests covering loaded AND unloaded origin-vessel mutation + save persistence.

**Explicitly NOT in M1:** loading/pickup direction and mixed windows (M3); cargo escrow / the `RouteStore` reservation map (M3 - M1 routes are single-window, the debit fires at the one dock-phase crossing via `EmitLoopCycle`, so there is no dispatch-to-debit gap to escrow); harvest provenance and full-run manifests for all runs (M2); mod-resource fixtures (M2); multi-stop/round-trip (M4); inter-body (M5); live route hold-state legibility text beyond the existing reason strings (M6). **Inventory debit is also deferred** (design decision D6 below): M1 debits resources only; a non-KSC route whose manifest includes inventory payload items holds, with the reason named in the log (the window shows the generic per-status text until M6). **The physical debit is loop-path-only** (design decision D11 below): the legacy non-loop `ApplyDispatch` path keeps v0 behavior byte-identical.

## 3. Design decisions

**Open question for the maintainer (only one):**

- **OQ1 - inventory debit deferral (D6).** M1 debits resources only. A non-KSC route whose `InventoryCostManifest` is non-empty will hold in `WaitingForResources` with reason `inventory-origin-debit-unsupported`. Rationale: 19.4 M1 names only the `PartResource.amount` / `ProtoPartResourceSnapshot` resource split; `GameAction` has no route inventory manifest field by deliberate deferral (`GameAction.cs:450-456`); removing exact `STOREDPART` payloads by identity from an origin is the M3 stock-slot-identity work; and delivering items without debiting them would duplicate matter, violating the conservation premise of the whole milestone. The alternative (full inventory mirror) adds an identity-matching removal writer + inventory availability probe, roughly doubling Phase 4/5.

**Decisions (chosen, with rationale):**

- **D1 - debit basis = `route.CostManifest`.** The origin is debited exactly what the route's recorded outflow says, which in v0 is the delivery manifest (`RouteBuilder.cs:380-382` copies it). The hub loses what the colony gains; the transport's own transit fuel was aboard at recording start and is not debited. Matches 19.4 ("compare against the route's recorded outflow") and scenario 2 (19.3).
- **D2 - apply pattern = delivery pattern, NOT a recalc applier.** Direct write at emit time inside `EmitDispatchDebit` + `RouteCargoDebited` row carrying actuals; `RouteModule` stays observe-only (T-ROUTEMODULE-OBSERVE); `FundsModule` continues to process only the `RouteKscFundsCost` component (0 for non-KSC, so non-KSC debits add nothing to the funds walk - already the case at `FundsModule.cs:244-251`). Revert safety: rewind restores vessel state from the RP quicksave; cutoff/tombstone-excluded rows vanish from ELS so the existing `(RouteId, RouteCycleId)` replay backstop (`EmitLoopCycle` :904) lets a re-flown timeline re-debit. See the discrepancy resolution in section 2.
- **D3 - all-or-nothing at the gate, clamp-and-warn at the writer.** `OriginHasCargo` requires `stored >= required` per resource (19.2.5 rule 1; the short resource is named). At debit-apply time (same tick, gate already passed) a mid-tick drift is clamped per-resource with a Warn and recorded as actual-vs-requested on the row - mirroring the delivery side's `RouteRequestedResourceManifest` convention rather than aborting a half-emitted cycle.
- **D4 - origin gate/writer share the delivery flow policy.** `ShouldDeliverToResource` (flowState true, not NO_FLOW) is reused for *reading and removing* from origin tanks: a flow-locked tank is player intent on the origin side too, and gate/writer symmetry (probe counts only what the writer may touch) is the established invariant (`LiveDeliveryCapacityProbe.cs:97-100`).
- **D5 - descriptor capture extends `OriginPartnerCandidate` + `RouteOriginProof`, excluded from the proof hash.** New fields ride the existing pure-resolver path (keeps `BuildStartRouteOriginProof` pure/testable); serialized additively and sparsely in `RouteProofCodec`; NOT added to `RouteProofHasher` so existing routes' `RouteProofHash` values stay stable (no spurious `SourceChanged`). The hash pins the witnessed transfer; coordinates are resolution metadata. Recorded data immutability: old captures simply lack the fields; `RouteBuilder` falls back to today's PID-only endpoint. `IsSurface` derives from the partner's situation (LANDED/SPLASHED - PRELAUNCH partners are already filtered out by the resolver, `RouteProofCapture.cs:90-91`).
- **D6 - inventory debit deferred** (see OQ1).
- **D7 - undocked-start rejection lives in the analysis engine.** Adding the origin-provenance check to `AnalyzeTree`/`AnalyzeRecording` (not just the near-miss list) converts today's "candidate that fails at create with `endpoint-missing`" into a proper near-miss with workflow guidance - strictly better UX and one source of truth. `RouteBuilder`'s `endpoint-missing` reject stays as the defensive backstop. Check ordering: after `HasEndpointProof`, before `HasMixedPickupDelivery` (the workflow error outranks manifest-level complaints). Known cost: existing eligible fixtures must gain origin fields (section 5).
- **D8 - priority semantics: lower value dispatches first; default 0; floor 0.** `cmp = priority.CompareTo(priority)`, then `NextDispatchUT`, then ordinal `Id` - exactly the spec's comparator. Sparse codec (omit 0). Note honestly in docs: for loop routes the real contention effect is the per-tick processing order of simultaneous dock crossings against a shared origin (loop routes do not dispatch off `NextDispatchUT`); `NextDispatchUT` remains in the comparator as the spec's deterministic mid-key.
- **D9 - minimal priority UI = detail-panel stepper.** A `[-] N [+]` row next to `DrawCadenceStepper` (`LogisticsWindowUI.cs:1237`), using the established deferred-mutation pattern (`pendingPriorityRoute`/`pendingPriorityValue` committed in `ApplyPendingActions`). No new column (the route table is already width-constrained; a sort-header refactor is out of scope).
- **D10 - harness shape = 3 layers** (per the todo's own mitigation note): (a) xUnit codec layer - already exists for routes; extended with priority round-trip and `RouteCargoDebited` actuals/requested round-trip; (b) tightened ORDERED multi-string source-text gates (assert `LoadRoutesFrom` index < `RevalidateSources` index inside `ParsekScenario.cs`; keep the save-side and tick gates); (c) in-game lifecycle tests for the live/unloaded debit + persistence (the only place `ProtoPartResourceSnapshot` writes and `GamePersistence.SaveGame` can be exercised). No Unity-shim harness: the todo's own estimate (~120 files affected) makes it a separate project, and layers (a)+(c) cover the actual M1 risk (state mutation correctness + persistence), which a shim would not cover better.
- **D11 - physical debit is LOOP-PATH-ONLY (plan-review finding 1).** `EmitDispatchDebit` gains a `bool applyPhysicalOriginDebit` parameter: `EmitLoopCycle` passes `true`; the legacy `ApplyDispatch` (:997) passes `false`, keeping legacy rows byte-identical (unconditional `CloneManifest(route.CostManifest)`, no physical write). Rationale: the legacy path fires at cycle start (not the recorded dock phase, spec 19.2.5 rule 2) and has no replay backstop; production v0 creates loop routes only. This also keeps `Tick_NonKscOrigin_CostManifestPopulated_FundsCostZero` (`RouteOrchestratorTests.cs:444-469`) green unchanged. Log a one-line Verbose on the legacy path when the route is non-KSC ("legacy dispatch path: physical origin debit skipped").
- **D12 - the origin-debit test seam covers resolve+probe+plan+write and returns the row inputs (plan-review finding 3).** `DeliveryApplierForTesting` (:968, `Action<Route, double, IRouteRuntimeEnvironment>`) replaces the whole delivery half INCLUDING its row emission, so it cannot be mirrored literally: `EmitDispatchDebit` itself constructs the debited row and needs the actuals. Seam shape: `internal static Func<Route, double, IRouteRuntimeEnvironment, OriginDebitOutcome> OriginDebitApplierForTesting` where `OriginDebitOutcome` is a small struct `(Dictionary<string,double> ActualDebited, Dictionary<string,double> RequestedOnShortfall, uint OriginVesselPid, bool Short, bool Unresolved)`; production code routes through a private method with the same signature. Rule: a resolution that returns `true` but `vessel == null` (both existing fake envs do this: `RouteOrchestratorTests.cs:84-94`, `RouteLoopDeliveryFireTests.cs:149`) counts as UNRESOLVED (Warn + zero actuals + full requested manifest), so xUnit non-KSC fixtures hit a defined branch.

## 8. Risks + invariants to re-verify during review

1. **KSC debited-row shape regression.** `EmitDispatchDebit` :1075 currently clones `CostManifest` unconditionally; the KSC branch must stay byte-identical (FundsModule reservation projection and existing tests read it). Pinned by `Crossing_KscOrigin_DebitedRowByteIdenticalToV0Shape`.
2. **Eligibility short-circuit cost.** `OriginHasCargo` now resolves the origin vessel and walks its parts ~1/s per blocked route (gate step 5 already resolves it once; step 6 resolves again). Acceptable (mirrors delivery's re-resolution), but verify no per-frame call path reaches it (`TryComputeSecondsToNextDockCrossing` does not).
3. **Unloaded proto writes while on rails.** Mutating `ProtoPartResourceSnapshot.amount` on an unloaded vessel must persist through `GamePersistence.SaveGame` and re-load (delivery side has field-proven this exact mechanism, `LiveDeliveryWriters.cs:258-264`); the in-game save-roundtrip test pins the debit direction. Watch the loaded/unloaded race: `isLoaded` captured once per debit, same as delivery (:1289 rationale).
4. **Partially-debited state across save/load.** Debit and delivery emit in the same tick under one cycleId, and `EmitLoopCycle`'s ELS backstop keys on the *delivered* row - a save written between the two halves (impossible within one tick, but a crash mid-tick) would leave a debit row without a delivery row; on reload the same cycleId re-fires and re-debits. Document as the known one-tick crash window (identical exposure to the v0 funds debit; the recovery-credit code comments :886-893 already discuss this class).
5. **Rewind/re-fly tombstones.** No new machinery: verify after a rewind that (a) the origin's tank state comes back with the quicksave, (b) the debited rows past cutoff leave ELS so re-fired cycles pass `IsDeliveryAlreadyInLedger`, (c) `FundsModule` sees `RouteKscFundsCost = 0` rows as no-ops in `ComputeTotalSpendings`.
6. **All-or-nothing is gate-only in M1** (single window). The 19.2.5 multi-source escrow is M3; nothing in M1 may reserve cargo between dispatch and window phase - assert in review that the debit only ever fires inside `EmitDispatchDebit` at the dock-phase crossing.
7. **Proof-hash stability.** The descriptor fields must not enter `RouteProofHasher` (pinned by `Hash_UnchangedByOriginDescriptorFields`) or every existing docked-origin route flips `SourceChanged` on load via `RouteStore.RevalidateSources` (`RouteStore.cs:591-592`).

9. **Surface-fallback misresolution.** With `IsSurface = true` origins, a destroyed depot could proximity-resolve to a *different* vessel within 500 m, which would then be debited. Same accepted behavior as destinations (`RouteOrchestrator.cs:36-46` rationale); note it in the design doc flip.
