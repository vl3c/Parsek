# Supply Route v0: Re-Founding on the Missions Subsystem

Authoritative implementation plan. Builders follow this document. The Missions subsystem is LOCKED; a Supply Route v0 is re-founded as three bound things over a route-owned backing Mission segment held by Logistics.

This plan resolves all six contradictions, three gaps, and the ordering issues surfaced in reconciliation. Where a draft workstream disagreed with the reconciliation, the reconciliation wins and the divergence is called out inline.

It also incorporates a clean-context Opus plan review (GO-WITH-FIXES). The review's three must-fixes are folded in and labelled inline: (#1) each loop crossing emits the FULL cycle transaction (origin/funds debit via the extracted `ApplyDispatch` body + delivery via `ApplyDelivery`, one cycleId), NOT delivery alone (delivery-alone never debits origin and trips `RouteModule`'s out-of-order guard) -- Phase 4; (#2) `RouteBuilder` clamps `DispatchInterval >= backing-mission span` so the loop-clock cadence equals the dispatch interval (one crossing == one cycle) -- Phase 4/5; (#3) the route's source set + `TransitDuration` are widened to the rendered `[root..undock]` path, not the leaf-only dock child -- Phase 1/5.

## Phase 4: Delivery on the Loop Clock (delivery workstream)

**Goal:** Replace the self-timer fire gate for loop-routes with a loop-clock crossing detector that, on each crossing, emits the FULL per-cycle transaction (origin/funds debit + delivery), reusing the existing `ApplyDispatch` + `ApplyDelivery` halves and the funds/ledger/ELS path verbatim. Decision-b's "route owns its dispatch clock" is satisfied HERE (crossing detector + `LastObservedLoopCycleIndex`), not by the Mission anchor.

**CRITICAL CORRECTION (Opus plan review, must-fix #1):** the per-cycle transaction is SPLIT across two appliers, not one. `ApplyDispatch` (`RouteOrchestrator.cs:386-460`) emits `RouteDispatched` + `RouteCargoDebited`, performs the origin physical debit + KSC funds charge, and writes `route.KscDispatchFundsCost` (`:412`); `ApplyDelivery` reads that field (`:680`) and emits only `RouteCargoDelivered`. Calling `ApplyDelivery` ALONE would (a) never debit origin / charge funds (breaks design invariants 2.2 #5/#6), and (b) trip `RouteModule.ProcessCargoDelivered`'s out-of-order guard (`if (state.DispatchedCycles == 0) { Warn; return; }`, `RouteModule.cs:231-237`). So a loop crossing MUST emit BOTH halves under one cycleId.

**Tasks:**
1. NEW `Source/Parsek/Logistics/RouteLoopClock.cs`: `TryGetRouteLoopState(LoopUnit unit, double currentUT, out double loopUT, out long cycleIndex, out bool isInInterCycleTail)` wrapping the verified `GhostPlaybackLogic.TryComputeSpanLoopUT(currentUT, phaseAnchorUT, spanStartUT, spanEndUT, cadenceSeconds, out loopUT, out cycleIndex, out isInInterCycleTail, schedule:null, loiterCuts:null)` (`GhostPlaybackLogic.cs:6940-6950`; v0 passes `schedule:null, loiterCuts:null`, so the span is uncompressed and `RecordedDockUT` maps directly into `[spanStart, spanEnd]`). **Consume the `isInInterCycleTail` out-param:** when dispatch period >= span (enforced in task 0 below) the clock parks at `spanEnd` in the tail; the crossing check ignores tail samples. XML-doc cadence==span vs cadence>span.
0. **(must-fix #2) Pin cadence == DispatchInterval.** The unit's `CadenceSeconds = max(LoopIntervalSeconds, span, MinCycleDuration)` (`MissionLoopUnitBuilder.cs:197`), so `cycleIndex` ticks on `max(DispatchInterval, span)`, NOT on `DispatchInterval`. To keep "one crossing == one dispatch cycle", `RouteBuilder` (Phase 5) MUST require/clamp `DispatchInterval >= backingMissionSpan` (= `undockUT - root.StartUT`) at creation, with a reject reason or clamp-up + Info log. This matches the design's "minimum interval = chain duration" (1.4). With `DispatchInterval >= span`, `CadenceSeconds == DispatchInterval` and the crossing index equals the dispatch index. ## Highest Residual Risks

1. **Backing-mission interval-key derivation correctness (post-undock trim).** A peel branch that straddles the undock instant, or structural-peel edge clamping (`MissionComposition` clamps peel UT into `[runStart, runEnd]`), can over- or under-trim. The window START must derive from the tree ROOT, not `SourceRecording`. Mitigation: prefer keying the trim on the Undock `BranchPoint`'s child leg ids over a pure `StartUT >= undockUT` scan; pin against ONE real dock/deliver/undock recording in-game before finalizing; do not guess-rewrite if a playtest disagrees, pin the concrete case first.
2. **Member-index alignment across the union and the orchestrator clock.** The route's backing-mission member indices are committed-list indices. Both the host Build and the orchestrator's `LoopUnit` build MUST source the exact same `RecordingStore.CommittedRecordings`-derived `committed` list (same frame). A mismatch silently indexes the wrong recording. Enforce by passing the single `committed` (and `trees`) snapshot into both; in-game tests assert recording IDENTITY, not just "a ghost appears."
3. **Loop-clock crossing under warp / reset.** ~1 Hz orchestrator vs multi-cycle warp jumps; activate/scene-reload/Rewind resets `LoopAnchorUT`. Fire once per tick + snap forward; ELS `(routeId, cycleId)` idempotency is the backstop. The cycleId sequence does NOT track `cycleIndex` 1:1 after a snap (by design). Persist `LastObservedLoopCycleIndex` so a mid-cycle save/reload does not double-fire.
