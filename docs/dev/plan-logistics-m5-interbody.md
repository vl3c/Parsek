## 1. Summary + scope, with the worked example

M5 activates inter-body routes: a route whose backing Mission carries a Missions-layer relaunch schedule (the same-parent zero-drift `MissionRelaunchSchedule`) or a re-aim window schedule (the cross-parent synodic `ReaimWindowSchedule`) dispatches its delivery on the SAME launch UTs the ghost renders on, and `Route.CadenceMultiplier` (N) becomes a MODULO on the scheduled-launch index (deliver every Nth scheduled window) instead of a multiplier on a fixed interval. Everything fails closed: a null-schedule same-body route stays behavior-identical to today (bytes on disk identical), and a re-aim classification that declines to faithful must not mis-fire or double-fire a delivery.

**Worked example (Kerbin -> Duna, the scenario-family-7 route).** The player flies: launch from KSC, park in LKO, transfer to Duna at the recorded departure, capture, dock at a Duna station at `recordedDockUT`, transfer Ore->LFO or whatever the manifest says, undock. Commit; create the route. The backing Mission classifies re-aim-supported (`ReaimClassifier`), the builder engages re-aim with a synodic window schedule (window k departs at `RecordedDepartureUT + k * synodic`, ~2.28 Kerbin years apart), compresses the parking loiter, and renders one ghost flight per window; the loop clock parks at spanEnd between windows (nothing in transit, ghost hidden in the gap).

- **N=1:** every synodic window is deliverable. The loop clock's dock crossing fires exactly when the rendered ghost reaches the recorded dock phase inside window k; the full cycle (dispatch + debit + delivery) emits under `cycle-{Completed+Skipped}`; the next delivery is one synodic period later. The UT of each delivery is the UT the player can watch the ghost dock at.
- **N=2:** window indices are anchored at the first fired window A (decision D3). Windows A, A+2, A+4 deliver; windows A+1, A+3 still RENDER (the ghost flies - the world looks busy) but the crossing is consumed silently: `LastObservedLoopCycleIndex` advances, no ledger row, no `SkippedCycles` bump, no hold (decision D4).

**In scope:** the route-side window-basis classification, the residual-modulo firing rule for both schedule mechanisms, offset anchoring + rebase, kind-flip (decline-to-faithful) safety, the multi-stop composition of the modulo gate, the Logistics-window basis label + next-dispatch T- for windowed routes, tests, docs.
**Out of scope:** any Missions edit; reviving the dormant v0.5 self-timer inter-body fields (`Route.DispatchWindowEpochUT` / `DispatchWindowPeriod`, finding F10); parallel shipments / fleets (M-MIS-11 item 7); rendering changes of any kind; the M6 legibility items except the two UI surfaces named in P3.

## 3. Decisions

**D1 - Route-side window-basis classification (new pure predicate; does not touch `CycleIndexKind`).** Add `RouteWindowBasis { FlatInterval, ZeroDriftSchedule, ReaimWindows }` + `RouteLoopClock.DeriveWindowBasis(unit)`:
- `unit.RelaunchSchedule != null` -> `ZeroDriftSchedule` (identical to `CycleIndexKind.Scheduled`);
- else `unit.ReaimSchedule` present + `Valid` + `unit.ReaimPlan` present + `Supported` -> `ReaimWindows`;
- else `FlatInterval`.
Rationale: F4 - the mmis11 kind typing deliberately classifies re-aim as Flat (truthful for index arithmetic); M5 needs the render-window semantics, which is a different question, answered by a different route-layer enum. `CycleIndexKind` and the emitter stay untouched. The basis is derived per tick from the resolved unit (the unit is already resolved per tick; no new build cost through the mmis11 cache).

**D2 - The ONE route-side cadence rule: residual modulo.** *Delivery fires on rendered windows whose window index is deliverable under `(windowIndex - anchor) % N_residual == 0`, where `N_residual` is the part of `CadenceMultiplier` NOT already consumed by the Missions-side build:*
- `FlatInterval` -> modulo OFF (N is fully consumed by the flat cadence `DispatchInterval = N x TransitDuration`; today's path, behavior-identical);
- `ZeroDriftSchedule` -> `N_residual = 1` (N is fully consumed Missions-side: `minSpacing = cadence = DispatchInterval` throttles the schedule, F3; `sIdx` indexes the THROTTLED launches, so ghost and delivery stay in lock-step at every N and a route-side modulo would double-apply N). Granularity note (review C3, intended - do not "fix" this later): the zero-drift generator places launches at faithful windows at least minSpacing apart, so at N=2 the spacing is "the next faithful window after 2 x interval", not exactly every 2nd window of the N=1 schedule; delivery still fires on every rendered launch, so lock-step holds;
- `ReaimWindows` -> `N_residual = N` (the build discarded `DispatchInterval` at `MissionLoopUnitBuilder.cs:558`, F3; the ghost renders EVERY synodic window; delivery fires on every Nth of those same rendered UTs).
This satisfies design 0.9 ("N x the backing mission's NATURAL relaunch period") in both regimes and the M5 goal ("dispatches on the SAME launch UTs the ghost renders on") in both: zero-drift thins windows so delivery UTs == rendered UTs trivially; re-aim delivers on a subset of rendered UTs. The modulo-skipped re-aim window is the "world looks busy, transfers nothing" precedent from blocked cycles - but silent and free (D4).
**Rejected alternative (recorded):** pass an N-independent base interval to `BuildMission` for windowed routes and modulo EVERYWHERE (uniform "ghost renders all windows"). Rejected because it (a) changes existing N>1 zero-drift scheduled same-body route behavior (render thins today; those routes exist and work), (b) breaks ghost/delivery lock-step (DEL-1 doctrine) on the zero-drift path, (c) introduces a build-input feedback loop (the interval passed would depend on the basis, which depends on the build), and (d) requires `RouteBackingMission.BuildMission` to become conditional, eroding the byte-identity proof. Note this REFINES the design-doc M5 sentence ("only the route-side modulo on that index"): the modulo is route-side only, but on the zero-drift mechanism the residual is 1 because the schedule throttle already implements every-Nth-window. Doc correction staged in P4.

**D3 - Offset anchoring.** New sparse persisted field `Route.WindowAnchorCycleIndex` (long, default -1, codec key `windowAnchorCycleIndex`, sparse-omitted at -1, mirroring `lastObservedLoopCycleIndex`, F11). Semantics: the window index of the FIRST window the route fired under `ReaimWindows` basis. When -1 and an owed crossing arrives on a `ReaimWindows` unit, adopt `anchor = dockCycleIndex` and treat that window as deliverable - so the first crossing after creation / activation / rebase ALWAYS delivers, and every Nth window after it does. Reset to -1 wherever `LastObservedLoopCycleIndex` resets: `RouteOrchestrator.TryActivate`, `RouteCadence.ApplyMultiplier` (alongside the existing rebase, `RouteCadence.cs:171-186`), and the `RouteBuilder` default. **Cadence-rebase amendment (review C4):** under `ReaimWindows` basis, `RouteCadence.ApplyMultiplier` must NOT reset `LastObservedLoopCycleIndex` to -1; it re-baselines it to the CURRENT dockCycleIndex instead (per-stop `LastFiredCycleIndex` snapped through the same value). The -1 reset makes the already-delivered current window owed again under a FRESH `cycle-{C+S}` id, which the ELS `(RouteId, cycleId, stopIndex)` backstop cannot suppress (`RouteOrchestrator.cs:771, :999`) - on re-aim that is a duplicate delivery for a window whose ghost does not re-fly, with the next real window ~2.28y away. Flat routes keep the accepted F5 one-immediate-fire reset unchanged; the anchor still resets to -1 at all three sites, so the first crossing after a rebase re-anchors and delivers. Test: `CadenceRebase_WindowedBasis_NoDuplicateDeliveryOfDeliveredWindow` (P2). Not consulted at all for `FlatInterval` / `ZeroDriftSchedule` (N_residual there is 1 or OFF).

**D4 - What a modulo-SKIPPED window does.** Advance `LastObservedLoopCycleIndex` to the skipped window's dock cycle index (and, multi-stop, snap every stop's `LastFiredCycleIndex` through it atomically, the blocked-cycle shape at `RouteOrchestrator.cs:1047-1080`) - and NOTHING else:
- NO `SkippedCycles` bump. `SkippedCycles` is the blocked-cycle counter with player-facing meaning and it feeds `cycleId = cycle-{Completed+Skipped}`; a modulo skip is scheduled behavior, not a failure. Safety holds: a skipped window writes NO rows, so leaving the cycleId counter unmoved cannot collide replay keys - cycleId uniqueness is required per FIRED cycle only (the M4a BLOCKER-1 safety property).
- NO hold (`RecordHold` is for blocks; the M6 hold line must not show a scary reason for a scheduled skip).
- NO ledger row, NO escrow reserve (the multi-stop skip happens before the dispatch guard / `ReserveCycleEscrow` at :1029/:1093-1102).
- NO partner-alternation advance and NO partner-gate satisfaction (review Missed #3): `AdvancePartnerAlternationOnDispatch` fires only on a genuine dispatch (`RouteOrchestrator.cs:836, :1117`), and the round-trip partner gate reads `partner.CompletedCycles` (`RouteDispatchEvaluator.cs:379`), which a skip never bumps - correct by construction, stated here so it stays that way. Benign consequence, documented so a playtester does not file it as a stall: a LINKED windowed route at N=2 makes its partner wait about 2 synodic periods for its turn - intended alternation behavior. Test: `ModuloSkip_DoesNotAdvancePartnerAlternation` (P2).
- YES flush of the prior cycle's pending recovery credit (`EmitPendingRecoveryCredit`), because the skip IS the "next crossing" for the previously dispatched cycle, mirroring the blocked path at :786 (OQ3, DECIDED: yes).
- One `Verbose` (rate-limited per route) log line: `LoopRoute: route {id} window {idx} SKIPPED by cadence modulo (N={n} anchor={a} basis={basis}) - marker advanced, nothing emitted`.

**D5 - Warp rule under the modulo.** When the owed dock cycle jumps K>1 windows in one tick, fire ONCE for the HIGHEST DELIVERABLE window index in `(lastObserved, dockCycleIndex]`; snap `LastObservedLoopCycleIndex` to `dockCycleIndex` either way; if no deliverable window is in the range, fire nothing and snap. This preserves the existing fire-once-snap-forward warp semantics (never replay skipped cycles) and never lets the modulo pick a window that is not deliverable. Pure helper: `ComputeHighestDeliverableWindow(lastObserved, dockCycleIndex, anchor, nResidual)`.

**D6 - Basis-flip safety (decline-to-faithful and back): cadence-rebase rules.** New sparse persisted marker `Route.ReaimWindowBasisEngaged` (bool, codec key sparse-omitted when false). Transition rules evaluated at the top of the loop path each tick, from the derived basis:
- **Engage** (marker false, basis `ReaimWindows`): set the marker; reset `WindowAnchorCycleIndex = -1`; and RE-BASELINE into the NEW (window) index space, symmetric to Decline: set `LastObservedLoopCycleIndex = dockCycleIndex - 1` (current window index minus one) and snap per-stop `LastFiredCycleIndex` through the same value. This leaves exactly the CURRENT window owed, so the first crossing after engage still delivers via D3 anchor adoption (the engage-must-not-suppress intent, preserved - the existing activate-reset contract already tolerates one immediate fire, F5). Why the re-baseline is REQUIRED (review C6, the blocker): flat-cadence indices are large (cadence = `DispatchInterval`, days) while synodic window indices are small (~2.28y cadence); after a transient decline (one build with degraded bodyInfo - `FlightGlobalsBodyInfo` degrades to no phase-lock on an empty `FlightGlobals`, worktree `RouteOrchestrator.cs:1716-1718`, and geometry digests are in the build signature, so the mmis11 cache does NOT suppress such a rebuild) followed by re-engage, a stale `LastObservedLoopCycleIndex` left in flat space exceeds every future synodic `dockCycleIndex`, `TryGetOwedDockCrossing` never emits again (worktree `RouteLoopClock.cs:339`), and the route silently never delivers - a PERMANENT SKIP the UT=0 cold-load gate does not cover. **Rejected alternative (recorded):** re-baseline to -1 instead; rejected because -1 marks every window since the phase anchor as owed and leans on the warp fire-once snap to collapse a synthetic multi-window jump, while `dockCycleIndex - 1` owes exactly the current window and matches D3's first-crossing-after-engage-delivers directly. Regression test: `DeclineThenReEngage_NextWindowStillFires` (P2, required-RED against the pre-review Engage shape that reset only the anchor).
- **Decline** (marker true, basis NOT `ReaimWindows` - the re-aim classifier / window planner declined this BUILD): RE-BASELINE with NO fire this tick: set `LastObservedLoopCycleIndex = dockCycleIndex` (current index in the NEW index space), reset per-stop fire state, clear the marker, reset the anchor, `Info`-log the transition. Rationale: the faithful flat index space (cadence = `DispatchInterval`) and the synodic window space are unrelated; a stale marker from one space compared in the other can look like a huge owed jump and fire a delivery the player never scheduled - the exact mis-fire the milestone forbids. A decline is a property of the BUILD, not of a window, so "decline on window k, valid on k+1, warp spans both" cannot occur within one tick; the re-baseline consuming the whole warp span with no fire is fail-closed by design (review C6). The route then behaves exactly like today's faithful cross-parent route (D7).
- **Build-level verdicts ONLY (review C6 secondary):** the basis and this evaluator key exclusively on the BUILD-level verdict (the resolved unit's `ReaimSchedule` + `ReaimPlan`, D1), never on `ReaimPlaybackResolver` PER-WINDOW declines (`ReaimPlaybackResolver.cs:336, :387, :396, :520`). Those are render-side only: they do not null the unit's `ReaimSchedule`, the basis stays `ReaimWindows`, and delivery still fires on that window's dock phase while the ghost flies faithful geometry.
- Flat <-> ZeroDrift flips (settings-driven, e.g. `TransitedBodyRotationMode`) keep today's behavior unchanged - they exist pre-M5 and the marker does not track them.
The transition evaluator is a pure `internal static` (`EvaluateWindowBasisTransition`) with full log-assertion tests. Flip thrash (a builder alternating verdicts across ticks) is bounded by the signature-gated mmis11 build cache; both re-baselines are idempotent (a second decline or engage tick is a no-op) and the log is rate-limited.

## 6. Replay / idempotency analysis

- **cycleId discipline.** `cycleId = cycle-{Completed+Skipped}` advances only when a cycle FIRES (Completed) or BLOCKS (Skipped). A modulo skip advances neither (D4) - safe, because the uniqueness invariant is per FIRED cycle (M4a BLOCKER-1 wording) and a skipped window writes zero rows; the next fired cycle reuses the id the skipped window never consumed, colliding with nothing in the ELS.
- **Dispatch-keyed backstop.** `IsDispatchAlreadyInLedger(routeId, cycleId, stopIndex)` (F6) still guards the fired path only; skips never reach it. A reload that re-presents a FIRED window (marker lost / stale) recomputes the same `cycle-{C+S}` (both counters persisted, `RouteCodec.cs:131-132`) and the guard suppresses the re-emit - unchanged from M4a.
- **Save/reload mid-synodic-gap.** Persisted: `LastObservedLoopCycleIndex`, `WindowAnchorCycleIndex`, `ReaimWindowBasisEngaged`, `Completed/SkippedCycles`, `LoopAnchorUT`. The window index is deterministic across reload: the reaim schedule rebuilds from recorded geometry + `referenceUT` derived from the persisted `LoopAnchorUT` floored to first-play (`MissionLoopUnitBuilder.cs:296-302`), giving the same `PhaseAnchorUT`/`CadenceSeconds`, hence the same `cycleIndex` at the same UT; the zero-drift schedule likewise (`FirstLaunchUT` from the same reference). In the parked inter-window tail the clock reports `loopUT == spanEnd >= dock`, so `ComputeDockCycleIndex` returns the CURRENT (already-consumed) cycle and `dockCycleIndex == lastObserved` - no crossing, no fire, whether or not the pre-save tick fired or skipped it. Residual risk (builder nondeterminism across versions/settings changing the window set) is covered by D6's decline/engage re-baseline plus the ELS backstop; add the cold-load guard note: the orchestrator must never evaluate transitions at `Planetarium UT == 0` (existing `IsCurrentUtReadyForCutoff`-style gate pattern; verify the loop path's UT source in P2).
- **Escrow.** RAM-only escrow is reserved only inside deliverable fired cycles (D10); modulo skips reserve nothing, so scene-change `ClearAllEscrow` / reload re-establish logic is untouched. `DropRouteEscrow` on pause and on cycle completion is unchanged.
- **Double-fire on decline.** The D6 decline re-baseline is the specific guard: without it, `lastObserved` from the synodic index space (small numbers, one per ~2 years) compared against a flat-cadence index (potentially large) produces an owed jump and one spurious fire; the `DeclineToFaithful_RebaselinesWithoutFire...` test is required to go RED against a no-D6 build before it goes green.
- **Permanent skip on re-engage.** The symmetric hazard (review C6, the blocker): without the D6 Engage re-baseline, a transient decline leaves `lastObserved` at a huge flat-space number, every future synodic `dockCycleIndex` stays at or below it, and `TryGetOwedDockCrossing` (worktree `RouteLoopClock.cs:339`) never emits again - the route silently never delivers. The `DeclineThenReEngage_NextWindowStillFires` test is required to go RED against the pre-review Engage shape before it goes green.
