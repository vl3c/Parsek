## 1. Summary + scope

M4 makes logistics SHAPE-general: a route can have N delivery/pickup stops fired at N recorded dock phases (multi-stop), load from N distinct sources gated all-or-nothing with cargo held in escrow between dispatch and each window's debit (multi-origin, 19.2.5), and chain to a partner route so an outbound + return pair alternate (round-trip linking). It lifts the `MultipleConnectionWindows` reject that today caps every route at one window.

**The single structural finding that reshapes the milestone (verified, all 7 readers):** the ANALYSIS / closure / codec layer is ALREADY list-shaped and multi-window-ready - `ComputeFlowClosure` sums over a window LIST (`RouteAnalysisEngine.cs:868`, pinned by `ComputeFlowClosure_MultiWindowSum_*`), `Route.Stops` is a real `List<RouteStop>` (`Route.cs:65`), `RouteStop` carries the reserved `SegmentIndexBefore = -1` / `DeliveryOffsetSeconds` (`RouteStop.cs:52-55`), and the `RouteCodec` STOP region loops on read AND write. But the FIRING / IDEMPOTENCY layer is single-dock/single-stop SCALAR: `RouteLoopClock` keys on one `recordedDockUT`, `Route.RecordedDockUT` / `Route.LastObservedLoopCycleIndex` are per-route scalars (`Route.cs:324,351`), `ResolveCycleStop` hardcodes `PendingStopIndex >= 0 ? PendingStopIndex : 0` (`RouteOrchestrator.cs:3066-3070`), the per-cycle ELS replay backstop keys only `(RouteId, cycleId)` (`IsDispatchAlreadyInLedger`, `:987`), and the ledger rows use FIXED `Sequence` 0/1/2. **So M4's real depth is the per-window FIRING + IDEMPOTENCY rework, NOT the analysis. The design doc's "codec cost is small / ComputeFlowClosure is a fill-in" framing is correct for the destination/window/closure axis but UNDERSELLS (a) the multi-window replay-key / cycleId / Sequence rework and (b) the multi-SOURCE Route shape (genuinely new data, not a reserved-field fill-in).**

**Recommended cut (subject to OQ1):** three ordered sub-slices, each its own correctness-complete shippable increment.

- **M4a - Multi-window foundation + multi-stop DELIVERY.** Lift the N-window reject, order windows by `DockUT`, widen `RouteAnalysisResult` to N stops, build N `RouteStop`s, build the per-window FIRING infrastructure (windowed replay key + per-stop fire state + `Sequence` uniquification + the cycleId decision - the four ranked blockers), fire delivery at EACH window's phase, resolve one endpoint per stop (reuse), end-trim the rendered window at the LAST dock, pin the no-spawn regression, and add the mid-recording-start rejection text. **No physical source debit per window, so NO escrow.** Ships scenario family 1/2 multi-stop delivery (5.5: KSC -> Base A -> Base B -> recover).
- **M4b - Source-side multi-window: multi-origin + cargo escrow.** Builds on M4a's firing. Adds the per-source Route shape (genuinely new data), all-or-nothing N-source gating in `OriginHasCargo` (first-short names the SOURCE), the `RouteStore`-owned RAM-only escrow map + its lifecycle clear + tombstone revert, and per-window SOURCE debit firing (reuse `ApplyPickupDebit` per window). This is the first slice where a dispatch-to-debit gap exists, so escrow first becomes load-bearing here (19.7 item 6). Ships scenario family 3 multi-window shuttle (refinery-load + station-deliver, one recording) and scenario 6 multi-origin consolidation (A + B -> station).
- **M4c - Round-trip linking.** Thin `LinkedRouteId` chain-constraint scheduler: route B holds until partner route A completes a cycle, with persisted alternation state. Builds on direction (M3a) + multi-window (M4a). Ships the Routine-Mission-Manager profile (outbound dock, return, recover - two linked one-way routes).

**Deferred explicitly (each surfaces in a rejection reason or documented limitation, never silently):**
- **Mid-recording START-trim shapes** (undock -> undock shuttle runs whose route begins INSIDE a pre-dock recording): hit the locked Missions layer's gap 1 (a dock is not a selectable interval boundary INSIDE a recording). This stays a DOCUMENTED LIMITATION surfaced in the rejection reason - NOT a Missions edit. The Missions-side lift is M-MIS-5, GATED after this milestone (todo doc).
- **Inter-body re-aimed multi-stop / multi-origin** (M5): a multi-stop run whose legs cross SOIs still rides the faithful path; re-aim activation is M5.
- **Parallel shipments / fleets** (M-MIS-11 item 7): several cycles of one route in flight at once needs per-INSTANCE marker crossings + per-instance escrow; park until a player profile demands it.
- **Per-stop destination-full gating across stops:** `DestinationHasCapacity` is a v0 true-stub (`LiveRouteRuntimeEnvironment.cs:233-252`); per-resource clamp at delivery time is unchanged. M4 does not add per-stop pre-gating.

**The M4-internal boundary, stated crisply:** M4a = N windows, fire DELIVERY at each phase, ONE physical-source debit at most (the M3a single bidirectional window still works; a SECOND source-debiting window defers to M4b with a clear reason). M4b = N source-debiting windows + N distinct sources + escrow. M4c = two routes, one chain constraint.

## 2. Current-state findings (verified against `1b014d629`)

All anchors below were verified by the understand workflow and re-spot-checked. **Path root: the analysis / orchestrator / writer / route / backing-mission surface all live under `Source/Parsek/Logistics/`**; `RouteProofMetadata.cs` sits under `Source/Parsek/`; Missions seam symbols (`MissionLoopUnitBuilder`, `MissionIntervalSelection`, `GhostPlaybackLogic.TryComputeSpanLoopUT` / `ShouldSpawnAtRecordingEnd`) under `Source/Parsek/`. Line anchors are illustrative (files drift); the mechanisms are the contract.

**The N-window reject fires at TWO entry points.** `AnalyzeTree` returns `MultipleConnectionWindows = 2` on the second `IsComplete` window (`RouteAnalysisEngine.cs:214`); `AnalyzeRecording` does the same (`:261`). Each dock-merged child recording carries EXACTLY ONE window (`ParsekFlight.cs` assigns `mergedChild.RouteConnectionWindows = new List<...>{ window }` per dock), so a multi-window route = multiple dock-merged child recordings across the tree, one window each. **`AnalyzeTree` iterates an UNORDERED `HashSet<string>` of recording ids** (`CollectSourcePathRecordingIds` returns a HashSet), so M4 MUST sort accepted windows by `RouteConnectionWindow.DockUT` (`RouteProofMetadata.cs:52`) - there is no existing chronological collector. `RouteConnectionWindow.IsComplete => !double.IsNaN(UndockUT)` (`:69`) is the accept gate; `DockUT` is the sort key.

**`ComputeFlowClosure` is already list-aware; the call site feeds length-1.** Signature takes `List<Dictionary<string,double>> loadedWindows` + `List<Dictionary<string,double>> deliveredWindows` (`RouteAnalysisEngine.cs:868`), unions names and sums each across the list (`SumWindowList`), rejects on `slack < -GainEpsilon`. The SOLE caller wraps the single window's manifests in `new List<...>{ ... }` (`:435-440`). **Multi-window is a call-site fill-in, NOT a method rewrite** (pinned multi-window-ready by `RouteAnalysisEngineTests.ComputeFlowClosure_MultiWindowSum_OverDeliveryAcrossWindows`). Two correctness terms for the run-level call: `residual` must be the LAST (run-end) window's `UndockTransportResources` (a sum double-counts cargo still aboard); `launched` stays the anchor leg's `StartTransportResources`; the M2 harvest "covered" term (`RouteHarvestAnalysis.cs`) must SUM all N windows' loaded terms or a multi-pickup run false-rejects as `UntrackedCargoGain`.

**`RouteAnalysisResult` is single-window-shaped (fan-out hazard).** All fields singular: `ConnectionWindow`, `ResourceDeliveryManifest`, `InventoryDeliveryManifest`, `ResourceLoadManifest`, `InventoryLoadManifest` (`RouteAnalysisEngine.cs:63-139`); `HarvestedManifest` / `IsHarvestOrigin` / `FirstHarvestWindow` are legitimately run-level singletons. Multi-window needs a per-stop collection (e.g. `List<RouteAnalysisStop>` carrying window + delivery + load + endpoint + DockUT). **Every consumer of `analysis.ConnectionWindow` / `analysis.ResourceDeliveryManifest` must be updated** (review-corrected list): `RouteBuilder`, `RouteCreationFormatters`, `RouteCandidateFinder`, AND `UI/LogisticsWindowUI.cs` (`candidate.Analysis.ConnectionWindow?.EndpointAtDock` `:1150`, `.ResourceDeliveryManifest` / `.InventoryDeliveryManifest` `:1163-1164`, `:1700` - a DISPLAY consumer the original list missed; because D2 keeps the singular scalar fields, this is a display-completeness risk - a multi-stop candidate row shows only the anchor window - not a build break). `RouteFundsCalculator` is NOT an analysis-result consumer (grep-confirmed zero `analysis.*` reads): the funds-under-N-stops question is Route-side (`ComputeDispatchFundsCostForRoute` walking `SourceRefs[0]` launch manifest), not analysis fan-out. Grep before touching the shape.

**`RouteBuilder` builds exactly one stop.** `var stop = new RouteStop { ... SegmentIndexBefore = 0, DeliveryOffsetSeconds = 0.0 }` then `Stops = new List<RouteStop>{ stop }` (`RouteBuilder.cs:451-487`); `recordedDockUT` / `transitDuration` / `DockMemberRecordingId` use the single window (`:134-167, 618-619`). M4 loops ordered windows, sets per-stop `SegmentIndexBefore` (member-recording index that triggers the stop) + `DeliveryOffsetSeconds` (= `window.DockUT - rootLaunchUT`), and picks the LAST dock for the route span.

**`RouteStop` / `Route` / `RouteCodec` reserve the destination multi-stop shape, NOT the source side.** `RouteStop` has `SegmentIndexBefore = -1` + `DeliveryOffsetSeconds` + the M3a `PickupManifest` / `InventoryPickupManifest` (`RouteStop.cs:34-55`). `Route.Stops` is a `List<RouteStop>` (`Route.cs:65`); the scheduler fields `CurrentSegmentIndex = -1` (`:151`), `PendingDeliveryUT` (`:156`), `PendingStopIndex = -1` (`:159`) are dormant (one-stop today). The codec STOP region loops on read and write, so N stops parse with zero structural change. **BUT `segmentIndexBefore` / `deliveryOffsetSeconds` are written NON-SPARSELY at placeholder 0 / 0.0** (`RouteCodec.cs:530-531`, unlike the sparse PICKUP_MANIFEST nodes) - changing how `RouteBuilder` derives them changes the bytes of new routes while old saves reload 0/0.0. **`Route` has ONE `Origin` + ONE `CostManifest` + ONE `InventoryCostManifest` (`:62,89,92`); there is no reserved per-SOURCE shape.** Multi-origin is genuinely new Route data + new codec, not a reserved-field fill-in.

**The per-cycle FIRING + IDEMPOTENCY layer is scalar (the four blockers).**
- `IsDispatchAlreadyInLedger(route.Id, cycleId)` keys only `(RouteId, cycleId)` (`RouteOrchestrator.cs:987`); `cycleId = "cycle-{CompletedCycles + SkippedCycles}"` (`:662`). The ledger ROWS already carry `RouteStopIndex` (delivery row `RouteStopIndex = ctx.StopIndex` at `:2400`; pickup row at `:1188`), but the dispatch/debit rows set `RouteStopIndex = -1` (`:1948, :1967, :2095`) and the KEY does not read it. So N windows under one cycleId: the FIRST window's row makes the check return true and silently suppresses windows 2..N forever.
- `Route.RecordedDockUT = -1.0` (`Route.cs:324`) + `Route.LastObservedLoopCycleIndex = -1` (`Route.cs:351`) are per-route scalars; firing at EACH window's phase needs per-stop dock-UT + per-stop observed-cycle tracking + codec + resets at all THREE `LastObservedLoopCycleIndex` reset sites (`RouteOrchestrator.TryActivate` `:190`, `RouteCadence.ApplyMultiplier` `:172`, `RouteBuilder` `:621` - NOT `Route.cs:186-192`, which is unrelated field declarations).
- `RouteOrchestrator` rows use FIXED `Sequence` 0 (dispatch) / 1 (debit) / 2 (pickup) / delivery; under N windows at near-equal UTs the `(UT, Sequence)` ordering any ledger walker relies on collides across windows.
- `ResolveCycleStop` hardcodes `PendingStopIndex >= 0 ? PendingStopIndex : 0` (`:3066-3070`). `RouteHasDeliveryManifest` (`:3035-3042`) / `RouteHasPickupManifest` (`:3051-3058`) are NOT per-route booleans (review correction): both call `ResolveCycleStop(route)` and read THAT resolved stop's manifest. They read per-route ONLY because `ResolveCycleStop` resolves a single stop via the default `PendingStopIndex = 0`. **The fix is to DRIVE `PendingStopIndex` to the firing window's index per crossing; the helpers then automatically reflect the right stop - no per-stop rewrite of the helpers themselves.**

**`RouteLoopClock` is single-dock.** All methods (`IsDockUTInSpan`, `ComputeDockCycleIndex`, `IsDockCrossing`, `TryComputeSecondsToNextDockCrossing`) take one `recordedDockUT` scalar (`RouteLoopClock.cs:94-202`). **`RouteEndpointResolver` is already per-endpoint (reuse).** `TryResolveEndpoint(RouteEndpoint, out Vessel, out reason)` is endpoint-parameterized (PID-first + 500 m surface-proximity fallback) (`RouteEndpointResolver.cs:41-93`). `RouteDispatchEvaluator.CheckEligibility` step-5 ALREADY iterates `route.Stops` resolving each `stop.Endpoint` and returns `stop-{i}-{reason}` on first miss (`RouteDispatchEvaluator.cs:175-189`) - **dispatch-time endpoint gating is already N-stop-correct.** Caveat: the resolver does bare-PID matching with no guid guard (`:49-60`), so two stops with the same craft-baked `persistentId` resolve to the same live vessel (OQ6).

**The dispatch gate order is fixed; all-or-nothing slots into the origin step.** `CheckEligibility` (`RouteDispatchEvaluator.cs:161-228`): ERS -> endpoint-per-stop -> `OriginHasCargo` (`:202-207`) -> Career+KSC funds -> `DestinationHasCapacity` (true-stub). `OriginHasCargo` resolves ONE `route.Origin` (`LiveRouteRuntimeEnvironment.cs:132`), captures ONE `originIsLoaded` (`:151`), gates ONE `route.CostManifest` (`:153`) + ONE `route.InventoryCostManifest` (`:186`) via `RouteOriginCargoCheck.HasRequired` / `HasRequiredInventory` (all-or-nothing, first-short by ordinal). M4b loops the per-source list here, and the M3a `ApplyPickupDebit(endpoint, manifest, env, routeId)` (`RouteOrchestrator.cs:1492-1601`, decoupled from Route in M3a) is the per-source debit to escrow against.

**`RouteStore` has NO escrow map and NO lifecycle reset hook.** It owns only `committedRoutes` (`RouteStore.cs:25`); `ResetForTesting` clears only that (`:124`); `RevalidateSources` / `SourceRefs` ERS machinery (`:266-595`) already iterates `route.SourceRefs` as a list (N-source ERS validation needs no change). There is no scene-change / session-reset hook - a RAM-only escrow map surviving a scene change with stale craft-baked pids would mis-gate.

**`LinkedRouteId` is serialized but ignored by dispatch.** `Route.LinkedRouteId` (`Route.cs:164`) is set null by `RouteBuilder.cs:611` and round-trips sparse (`RouteCodec.cs:112-113, 279-281`); grep confirms NO dispatch path reads it. `RouteStatusPolicy` has THREE switches that all throw-on-default (`RouteStatusPolicy.cs:65, 95, 128`) - a free compile gate forcing any new `RouteStatus` to be classified in all three. `CompareRoutesForTick` (priority -> NextDispatchUT -> ordinal id, `RouteOrchestrator.cs:310-400`) is total/NaN-safe and operates on a snapshot copy (no ordering change needed for round-trip).

**The no-spawn guarantee is architectural (reuse + pin).** The engine intercepts loop-unit members at `GhostPlaybackEngine.cs:1130` (`if (currentLoopUnits.TryGetUnitForMember(i, out unit)) { UpdateUnitMemberPlayback(...); continue; }`) BEFORE any past-end / `PlaybackCompleted` path, and `UpdateUnitMemberPlayback` never emits `needsSpawn`. Route-excluded recordings never become unit members (`RouteBackingMission.ComputeExcludedIntervalKeys` -> `MissionIntervalSelection.ComputeRenderWindows`). `ComputeExcludedIntervalKeys` (`RouteBackingMission.cs:83-161`) trims at/after ONE `segmentEndUT` (boundary `segmentEndUT - 1e-6`); the M-MIS-9 freeze re-derives against the single `Route.RecordedDockUT` (`:438-447`). **The walk does NOT internally know N docks - the generalization is the CALLER passing `max(stop.DockUT)`.** `BuildMission` runs every render frame (`:626-686`), so the widened exclusion-key set must be cached on the Route, not recomputed per frame.

**The Missions-boundary limitation is a hard, verified limit.** In always-tree mode a dock ALWAYS splits recordings, so END-trims `[launch..dockN]` are expressible (the merged child is a separate recording/interval). But a dock MID-recording is not a selectable interval boundary (`MissionIntervalSelection` renders a contiguous `[min-start, max-end]` window per vessel, `MissionIntervalSelection.cs:16-21`), so an undock -> undock shuttle whose run begins inside a pre-dock recording cannot be START-trimmed. ## 3. Design decisions + open questions

**Open questions. OQ1 + OQ2 DECIDED 2026-06-15 (maintainer: "as recommended"). OQ3-OQ8 DECIDED 2026-06-15 by a clean-context architecture analysis (verified against the code, file:line) that confirmed the firing/abort/round-trip recommendations and IMPROVED three: OQ5 (no new `Route` field - derive sources from pickup `Stops`, overturning RANK-7's "new persisted data"), OQ4 (no `ROUTE_DELIVERY_FAILED` type exists - degrade like the pickup-unresolved path), and OQ3 (defer the route-level cycle-snap so a partial cycle resumes window B after reload). - **OQ1 - The M4-internal cut. DECIDED 2026-06-15: as recommended.** M4a (multi-stop delivery, no escrow) -> M4b (source-side multi-window: multi-origin + escrow) -> M4c (round-trip). Rationale: multi-stop DELIVERY (launch cargo, funds at dispatch, deliver at each dock) opens no dispatch-to-debit gap, so it ships the whole multi-window FIRING foundation WITHOUT escrow; escrow lands with M4b where the first source-debit-at-window-phase gap exists (19.7 item 6). Alternative: fold escrow into M4a (build it for single-source multi-window pickup immediately) - more shipped per slice but couples two hard subsystems. **This is the headline decision; the rest of the plan assumes the recommended cut.**
- **OQ2 - Multi-window FIRING shape + the two replay guards (THE gating decision for Phase A3; review-confirmed blocker). DECIDED 2026-06-15: Horn A (one cycleId per cycle, N windows under one `EmitLoopCycle`, re-key BOTH guards, move the `CompletedCycles` bump to once-per-cycle).** The plan author must make D5/A3 internally consistent with Horn A. **There are TWO `(RouteId, cycleId)`-keyed replay guards on the loop path, not one** (review correction): `IsDispatchAlreadyInLedger` (`RouteOrchestrator.cs:987`, RouteDispatched-keyed) AND `IsDeliveryAlreadyInLedger` (`:2172`, RouteCargoDelivered-keyed) called INSIDE `ApplyDelivery`, which ALSO bumps `CompletedCycles += 1` on its replay branch (`:2188`). Its docstring says "dead for loop routes" - true ONLY because a loop route delivers ONCE per cycle today; under N delivery windows it is live and traps. `CompletedCycles` is bumped at THREE structural sites inside a cycle: `ApplyDelivery` (`:2416`), the pure-pickup branch (`:1105`), and the dispatch-replay branch (`:995`). `cycleId = "cycle-{CompletedCycles + SkippedCycles}"` is computed at tick start (`:666`) AND recomputed inside `ApplyDelivery` (`:2159`), so any mid-cycle `CompletedCycles` bump shifts the cycleId later windows compute. **The two firing-shape horns:**
  - **Horn A (recommended): ONE cycleId per cycle, N windows fire under one `EmitLoopCycle`.** Re-key BOTH `IsDispatchAlreadyInLedger` AND `IsDeliveryAlreadyInLedger` to `(RouteId, cycleId, stopIndex)` (both row types already serialize `RouteStopIndex` - delivery at `:2400`, so this reads an existing field). Move the `CompletedCycles` bump OUT of the per-window path (`:2416`, `:2188`) to a single "cycle complete after the last window fired" site; keep `ApplyDelivery`'s internal cycleId recompute (`:2159`) in sync with the tick-start computation (`:666`). Uniquify `Sequence` per window as `stopIndex * SEQ_STRIDE + intraWindowSeq` (SEQ_STRIDE >= 4).
  - **Horn B: each window fires as its own `EmitLoopCycle` crossing (per-window cycleId).** Then the once-per-cycle KSC dispatch/funds charge (5.1, ONE funds charge per multi-stop cycle) must be GATED to fire only at window 1, or it re-charges per window.
  **Either horn is a real restructure of `EmitLoopCycle` / `ProcessLoopRoute`** (today: ONE crossing on the single `route.RecordedDockUT` `:621-623`, ONE `EmitLoopCycle` call `:714`), NOT a "extend a key, read an existing field" one-liner. Reject the implicit "cycleId advancing mid-cycle" variant - it de-syncs the dispatch/deliver pairing the `RouteModule.ProcessCargoDelivered` `DispatchedCycles==0` guard (`:334`) relies on. A3 tests MUST exercise the REAL `ApplyDelivery` path (not the `DeliveryApplierForTesting` fake at `RouteLoopDeliveryFireTests.cs:80-90`, which bypasses `IsDeliveryAlreadyInLedger` and would go false-green).
- **OQ3 - Per-stop fire-state persistence. DECIDED (clean-context analysis, 2026-06-15).** Add a per-stop `long LastFiredCycleIndex = -1` AND a per-stop `double RecordedDockUT = -1.0` on `RouteStop`, sparse-coded inside the existing `STOP` node (omitted at -1, mirroring the route-level `lastObservedLoopCycleIndex` sparse write at `RouteCodec.cs:153`). **Drop the parallel-array alternative** (it couples array length to `Stops.Count` and needs its own index-alignment invariant). `Route.LastObservedLoopCycleIndex` STAYS the route-level span-cycle tracker (gates whether the cycle is observed); the per-stop `LastFiredCycleIndex` is the per-window sub-gate (`loopUT >= stop.RecordedDockUT && stop.LastFiredCycleIndex < dockCycleIndex` fires once, then sets it). **Structural requirement the first draft missed:** the route-level `LastObservedLoopCycleIndex` snap-forward (`ProcessLoopRoute:722`) must be DEFERRED until `loopUT >= max(stop.RecordedDockUT)` (recommended) - else a tick landing between dock A and dock B sees `IsDockCrossing` false and window B never fires; deferring the snap keeps one code path and preserves warp-jump-once. Resets at all THREE `LastObservedLoopCycleIndex` sites must also reset the per-stop indices: `RouteOrchestrator.TryActivate` (`:190`), `RouteCadence.ApplyMultiplier` (`:172`), `RouteBuilder` (`:621`, where the per-stop -1 default suffices but the A3 test asserts it). Byte-identity: a single-stop route's STOP node carries `segmentIndexBefore` + `deliveryOffsetSeconds=0` and NO `recordedDockUT`/`lastFiredCycleIndex` keys. The A3 test "window A fires, reload before B's phase, B fires on next tick" pins the deferred-snap resume.
- **OQ4 - `CompletedCycles` / abort semantics under N windows. DECIDED (clean-context analysis, 2026-06-15).** A cycle completes (bumps `CompletedCycles` ONCE) after the last window's phase fires, under Horn A. **All-or-nothing is at DISPATCH (the source gate), NOT mid-cycle delivery.** **There is NO `ROUTE_DELIVERY_FAILED` GameActionType** (verified) - a per-stop endpoint loss mid-cycle Warn-logs and emits NOTHING for that stop (mirroring the pickup-unresolved path: `ApplyPickupDebit:1535` records zero actuals + `Unresolved=true`, the caller `EmitPickupHalf:1146` Warn-logs and emits the row), does NOT transition the route, and records a per-stop hold detail. The route flips to `RouteStatus.EndpointLost` ONLY when EVERY stop is unresolvable (the single-stop case preserves v0's terminal `EmitEndpointLostAction` at `ApplyDelivery:2218`). `CompletedCycles` bumps exactly once for a partial cycle (not per surviving window); `SkippedCycles` is for the all-or-nothing DISPATCH block only (`ProcessLoopRoute:688`), never a mid-cycle per-stop loss. Bring the delivery side into parity with the pickup-unresolved degrade so one lost stop does not kill the surviving stops (replay fidelity: a witnessed delivery to A is not discarded because C moved). Note: `LastHold*` is route-level single-slot (`Route.cs:217-242`), so a multi-stop route surfaces only the last hold - accept for M4, document.
- **OQ5 - Multi-source Route shape (M4b, gates RANK-7). DECIDED (clean-context analysis, 2026-06-15): do NOT add a new `Route` field.** **Derive the source set from the pickup `Stops` at gate time, keyed by resolved live pid; the ONLY genuinely-new datum is the RAM escrow map (OQ7), not a source list.** The data already exists: each pickup window is a `RouteStop` with `Endpoint` + `PickupManifest` (`RouteStop.cs:13,34`) + `InventoryPickupManifest` (`:46`), and M3a's `ApplyPickupDebit(endpoint, manifest, env, routeId)` (`RouteOrchestrator.cs:1492`), `LiveOriginCargoProbe(Vessel, isLoaded)` (`LiveOriginCargoProbe.cs:35`), and `RouteOriginCargoCheck.HasRequired(manifest, reader, ...)` (`:27`) are ALL already per-source callable. ## 5. Gameplay grounding (the headline requirement)

Every M4 shape is grounded in a concrete player scenario: what the player physically records, what the route does per cycle, the economics (debit/credit/funds), and what is REJECTED with the player-language reason.

### 5.1 Multi-stop delivery (M4a) - scenario 1/2, design 5.5

- **Records:** Launch from KSC. Fly to Base A on the Mun, dock, transfer 150 LF + 183 Ox to Base A, undock. Fly to Base B on the Mun, dock, transfer 200 MP to Base B, undock. Deorbit and recover on Kerbin (or leave parked).
- **Per cycle:** At dispatch (Career, KSC origin) charge the KSC funds cost ONCE, on the LAUNCH-manifest basis (M2 decision) - the full launch load covering both deliveries. The ghost flies launch -> dock A. At dock A's recorded phase, deliver 150 LF + 183 Ox to Base A, clamped to Base A's live capacity (per-resource independent; surplus wasted, recorded requested-vs-actual). The ghost continues to dock B. At dock B's phase, deliver 200 MP to Base B, clamped. The ghost flies the return leg and ends (Recovered disposition credits stock recovery valuation; NO real vessel spawns - the no-spawn pin guarantees this).
- **Economics:** ONE funds charge at dispatch (KSC origin); TWO physical destination credits (no funds); recovery credit if recovered. No per-source debit (launch cargo).
- **Rejected:** transport still docked at recording end -> "Transport must undock from destination" (existing). A window that transferred nothing is not a stop. Flows don't close (more aboard than launched + harvested) -> `FlowDoesNotClose` naming the unaccounted resource + quantity. Two transfers at the same recorded UT -> `MultipleConnectionWindows` ("can't be ordered"). A run that STARTS between two docks (undock -> undock shuttle, no launch / dock-start) -> `MidRecordingStartTrimUnsupported` ("Routes must begin at launch or while docked to the origin; mid-flight start points aren't supported yet").

### 5.2 Multi-window shuttle (M4b) - scenario family 3, station<->refinery, one recording

- **Records:** A tug lives docked at a station. It undocks, flies to a refinery, docks, LOADS 400 LF from the refinery onto the tug (pickup), undocks, returns to the station, docks, DELIVERS 400 LF to the station, undocks, parks.
- **Per cycle:** At dispatch, gate the refinery source all-or-nothing (does the refinery hold 400 LF to give?). RESERVE 400 LF against the refinery in escrow. The ghost flies station -> refinery; at the refinery's window phase, physically DEBIT 400 LF from the refinery (loaded or unloaded), RELEASE the escrow. The ghost flies refinery -> station; at the station's window phase, CREDIT 400 LF to the station (clamped). Between dispatch and the refinery debit, the escrow stops a competing route from draining the refinery and stranding this cycle (19.2.5).
- **Economics:** the refinery loses 400 LF per cycle (physical debit, NOT funds); the station gains it (clamped). No KSC funds (non-KSC origin). A crashed disposition still debits the refinery (the outflow was witnessed; the cargo's fate is the route's loss).
- **Rejected:** refinery dry at dispatch -> `WaitingForResources` naming the refinery. Refinery vessel gone -> `EndpointLost`. A stored part on the tug the destination never gave it (unwitnessed inventory gain) -> `MixedPickupDelivery` (M3a).

### 5.3 Multi-origin consolidation (M4b) - scenario 6, A + B -> station

- **Records:** A transport launches, docks at Depot A, loads 100 ore, undocks, docks at Depot B, loads 200 ore, undocks, docks at the orbital station, delivers 300 ore, undocks.
- **Per cycle:** At dispatch, gate BOTH sources all-or-nothing - Depot A must cover 100 AND Depot B must cover 200, else hold in `WaitingForResources` naming the FIRST short source. RESERVE 100 against A and 200 against B. At A's window phase, debit 100 from A + release A's escrow; at B's window phase, debit 200 from B + release B's escrow; at the station's window phase, credit 300 (clamped). 
- **Economics:** A loses 100, B loses 200, station gains 300 (clamped); no funds (non-KSC). Escrow guarantees that if a competing route also draws from Depot A, it sees A's 100 already reserved and cannot steal it mid-run.
- **Rejected:** either depot short at dispatch -> `WaitingForResources` naming that depot. Flows don't close -> `FlowDoesNotClose`.

### 5.4 Round-trip with vehicle reuse (M4c) - Routine-Mission-Manager profile

- **Records:** Route A's recording: launch from KSC, dock at the station, deliver supplies, undock. (The player flies the transport home and recovers it = a SECOND recording.) Route B's recording: the return leg (station -> Kerbin, recover); a return that picks up cargo (e.g. science/empty containers) docks-loads at the station first.
- **Per cycle (linked A->B):** A dispatches, delivers at the station, completes. B was HOLDING ("waiting for partner route A"); now A completed, B dispatches, (optionally picks up at the station, debiting it), flies home, and the recovery credit values the onboard cargo (KSP stock recovery valuation). B completes; A becomes eligible again. The pair alternate so only one transport's worth of effects is in flight at a time (the conceptual single reused vehicle).
- **Economics:** A charges KSC funds + delivers; B picks up (debits the station) + earns recovery credit. The link is a scheduling constraint, not a new resource path.
- **Rejected:** partner paused -> the other dispatches on its own schedule (10.14); both linked and neither seeded -> the deadlock guard seeds the link-seed route first.
