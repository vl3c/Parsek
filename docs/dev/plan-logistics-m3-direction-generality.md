## 1. Summary + scope

M3 makes logistics direction-general: cargo can flow FROM an endpoint ONTO the transport (the reverse of v0's deliver-from-transport direction), one dock window can carry BOTH directions, and route validity becomes the single flow-closure rule (19.2.4) instead of the assume-start-loaded delivery heuristic.

**Two findings from the code map reshape the milestone before a line is written:**

1. **Connection-window capture is ALREADY fully symmetric - pickup needs ZERO recorder change.** `RouteConnectionWindow` (`RouteProofMetadata.cs:49-98`) already carries all four scoped corners - Dock/Undock x Transport/Endpoint - for resources AND inventory, captured symmetrically at the dock merge (`RouteProofCapture.BuildDockRouteConnectionWindow`, `:533-624`) and the undock split (`CompleteRouteConnectionWindowAtUndock`, `:660-724`). The recording-side proof hash (`RouteProofHasher.cs:82-129`) already hashes all four manifests. So a pickup window is witnessed by data that exists today; M3 bullet 1 ("source-side connection capture, mirror of delivery capture") is ALREADY DONE on disk. The pickup quantities (`endpointLoss = Dock - Undock` endpoint, `transportGain = Undock - Dock` transport) are even already computed in `HasResourcePickup` / `HasInventoryPickup` (`RouteAnalysisEngine.cs:491-556`) - but only to REJECT (`MixedPickupDelivery`, status 4). M3 turns those exact expressions from a reject into a manifest BUILDER. Consequence: pre-M3 recordings stay byte-identical for free (they carry no pickup windows), and most of M3 is analysis + applier + route-shape work, not capture work.

2. **True multi-origin (A + B -> station) needs N windows, which is M4.** The design doc lists multi-origin (19.2.5) under M3's final bullet, but a single connection window has exactly one endpoint partner; loading at depot A and depot B is two loaded windows, and a second completed window is rejected today (`MultipleConnectionWindows`, status 2) at BOTH analysis entry points (`RouteAnalysisEngine.cs:138-146`, `:188-192`) - lifting that reject is the M4 multi-window acceptance. The all-or-nothing source GATE and the escrow reservation MAP can be BUILT in M3 as infrastructure, but they have nothing to gate (one source) and no dispatch-to-debit gap to escrow (single-window debits AT the dock crossing, 19.7 item 6) until multi-window lands. This is the headline scope decision (OQ1).

**Recommended cut (subject to OQ1):**

- **M3a - Pickup direction + flow closure (single bidirectional window).** Self-contained, the pickup headline. Ships scenario 5 (load at a depot, fly home, recover with cargo) and scenario 3 (shuttle/tanker, loaded mid-run), plus mixed deliver-and-pickup at one dock. No recorder change. This is the M3 deliverable.
- **M3b - Multi-origin all-or-nothing + cargo escrow.** RESEQUENCE to land alongside M4 multi-window, where it is first exercisable. Not a scope cut (the work still happens); a move to where it can be tested against real geometry instead of a synthetic 2-window fixture.

**M3/M4 boundary, stated crisply (kept in the rejection text):** M3 = ONE window, TWO directions, ONE dock crossing. M4 = N windows / N stops, each firing at its own recorded phase. M3 keeps `RouteStop` length 1, `RecordedDockUT` single, `RouteLoopClock` single-dock-keyed, and the `MultipleConnectionWindows` reject in place. `SegmentIndexBefore` / `DeliveryOffsetSeconds` (`RouteStop.cs:24-31`) stay reserved/unused.

**Deferred explicitly:** N>1 connection windows and multi-stop firing (M4); round-trip linking (M4); inter-body re-aimed pickup routes (M5); full-run INVENTORY closure across legs if OQ3 keeps it window-local; player-facing multi-origin and the escrow map (OQ1 -> M4). Each deferral surfaces in a rejection reason or a documented limitation, never silently.

## 2. Current-state findings

All claims verified against this worktree by the understand-phase code map AND a clean-context plan review (2026-06-14, 3 reviewers, 0 blockers; corrections folded in below). **Path root: the analysis/orchestrator/writer/route surface lives under `Source/Parsek/Logistics/`** (`RouteAnalysisEngine.cs`, `RouteOrchestrator.cs`, `RouteDispatchEvaluator.cs`, `LiveRouteRuntimeEnvironment.cs`, `LiveOriginCargoProbe.cs`, `LiveOriginDebitWriters.cs`, `RouteOriginDebitPlanner.cs`, `RouteHarvestAnalysis.cs`, `Route.cs`, `RouteStop.cs`, `RouteStore.cs`, `RouteCodec.cs`); only `RouteProofMetadata.cs` / `RouteProofCapture.cs` / `RouteProofCodec.cs` / `RouteProofHasher.cs` sit directly under `Source/Parsek/`, and `Recording.cs` / `ParsekFlight.cs` / `GameActions/*` where named. Line anchors are illustrative (the files drift); the mechanisms are the contract.

**Capture is symmetric and immutable; pickup is an analysis re-interpretation.** `RouteConnectionWindow` stores raw per-phase amounts, not signed deltas: `DockTransportResources` / `UndockTransportResources` / `DockEndpointResources` / `UndockEndpointResources` plus the four matching inventory lists (`RouteProofMetadata.cs:58-65`), each scoped by `TransportPartPersistentIds` / `EndpointPartPersistentIds` (`:56-57`). Delivery = `endpointGain (Undock-Dock endpoint)` AND `transportLoss (Dock-Undock transport)`; pickup = `endpointLoss (Dock-Undock endpoint)` AND `transportGain (Undock-Dock transport)`. The dock baseline is the PRE-COUPLE snapshot pair (`pendingDockSelfSnapshot` transport / `pendingDockPartnerSnapshot` endpoint, `ParsekFlight.cs` OnPartCouple) so same-frame crossfeed cannot deflate the transport baseline; the undock baseline is the post-split pair verified disjoint by `TryVerifyRoutePartSetsSeparated`. **No new capture field, no new snapshot, no recorder hook for resource pickup.**

**The pickup arithmetic already exists, reject-only.** `HasResourcePickup` (`RouteAnalysisEngine.cs:491-524`) computes `endpointLoss` / `transportGain` and returns true (-> `MixedPickupDelivery`, stop) if either exceeds 1e-9. `HasInventoryPickup` (`:526-556`) does the same per `InventoryPayloadItem` identity. `HasMixedPickupDelivery` (`:466-474`) is the gate caller, run from `AnalyzeWindow` (`:262-273`). `BuildResourceDeliveryManifest` (`:558-610`, `delivered = min(endpointGain, transportLoss)` per routable name, undefined names excluded at `:589`) is the exact mirror template for the load-direction builder.

**M1 reuse is deep - the reverse-flow physical machinery is mostly built.** `LiveOriginCargoProbe.ProbeResourceStored` (`LiveOriginCargoProbe.cs:54-116`) already answers "does this vessel hold enough to GIVE?" (sums stored `amount` over deliverable tanks, loaded/unloaded split, `ShouldDeliverToResource` flow gate symmetric with the writer) - it IS the source-availability probe, just pointed at the origin today. `LiveOriginDebitWriters.WriteResourceDebit` (`LiveOriginDebitWriters.cs:52-114`, `pr.amount -= delta` loaded / `prs.amount -= delta` unloaded, clamp-at-zero) is the REMOVE writer - delivery's sign flip is already shipped. `RouteOriginDebitPlanner.PrepareDebit` (`available = min(required, stored)`) plans it. `ApplyOriginDebit` (`RouteOrchestrator.cs:1109-1207`: resolve vessel, capture `originIsLoaded` ONCE, probe, plan, write, return actual/requested/pid) is the applier shape. **Both `LiveOriginDebitWriters` and `LiveOriginCargoProbe` are resources-only; inventory debit is explicitly deferred to M3** (class doc `LiveOriginDebitWriters.cs:14-15`).

**Analysis is assume-start-loaded; M2 built the gain-side closure as the template.** `BuildResourceDeliveryManifest` reads ONLY the single window's four corners - launched provenance, between-window consumption, and end residual are never in the equation. M2 added `RouteHarvestAnalysis.CheckTransportGains` (`RouteHarvestAnalysis.cs:149-267`), engaged ONLY when every transport-lineage leg carries a COMPLETE `RouteRunCargoManifest` (start scope + `StartTransportResources` + `EndCaptured`), else `LegacyFallback` and the pre-M2 path runs byte-identical. It resolves the lineage + anchor (`FindGainAnchorIndex`), reads full-run Start/End manifests, admits witnessed gains, and rejects an uncovered gain as `UntrackedCargoGain` (status 7) with a `RejectDetail` naming resource+quantity through the formatter/near-miss surface. This lineage+anchor+full-run-manifest machinery is the template M3 extends from the one-sided gain inequality to the two-sided closure equality.

**Dispatch gate order is fixed; all-or-nothing slots into the origin step.** `RouteDispatchEvaluator.CheckEligibility` (`RouteDispatchEvaluator.cs:161-228`): ERS -> endpoint-per-stop -> `OriginHasCargo` (single call, `:202-207`) -> Career+KSC funds -> `DestinationHasCapacity` (true-stub; partial capacity clamps at `RouteDeliveryPlanner.PrepareDelivery`, zero capacity blocks `DestinationFull`). `OriginHasCargo` (`LiveRouteRuntimeEnvironment.cs:94-186`) gates ONE `route.Origin` vs ONE `route.CostManifest` via `RouteOriginCargoCheck.HasRequired` (all-or-nothing, first-short by ordinal name). It HOLDS a non-KSC route with a non-empty inventory cost manifest as `inventory-origin-debit-unsupported` (`:127-136`) - the explicit M1->M3 carve-out the inventory pickup writer must lift.

**Orchestrator fires both halves at one crossing; the replay backstop has a pickup-only hole.** `ProcessLoopRoute` (`RouteOrchestrator.cs:569-740`) detects the dock crossing (`RouteLoopClock.IsDockCrossing`), checks eligibility, then `EmitLoopCycle` (`:960-1036`) runs the per-cycle transaction under one `cycleId`: (1) flush prior recovery credit; (2) **ELS replay backstop `IsDeliveryAlreadyInLedger` keyed `(RouteId, RouteCycleId)` over the `RouteCargoDelivered` row** (`:981-994`); (3) `EmitDispatchDebit(..., applyPhysicalOriginDebit:true)` (debit half, `RouteDispatched` Seq0 + `RouteCargoDebited` Seq1); (4) arm recovery credit; (5) `ApplyDelivery` (credit half, `RouteCargoDelivered` Seq0, bumps `CompletedCycles`). The legacy self-timer path (`ApplyDispatch`, `applyPhysicalOriginDebit:false`) never touches tanks (M1 D11) and is dead for loop routes. **`RouteModule` is observe-only (T-ROUTEMODULE-OBSERVE); physical writes happen live at emit, reverted by the rewind quicksave, never by the recalc walk.**

**Route/RouteStore/RouteStop have no pickup or multi-source shape.** `Route` has one `Origin` + one `CostManifest` + one `InventoryCostManifest` (`Route.cs`). `RouteStop` has `DeliveryManifest` + `InventoryDeliveryManifest` only. `RouteStore` holds only `committedRoutes` (a `List<Route>`) + CRUD + codec drivers + the ERS-gated `RevalidateSources`; **no escrow / reservation map exists anywhere** (grep confirmed). `RouteRunCargoManifest` is resources-only (`RouteProofMetadata.cs:160` "No inventory fields in M2 - deferred to M3").

**Inventory identity is captured; no inventory-REMOVE writer exists.** `InventoryPayloadItem` (`RouteProofMetadata.cs:24-47`) carries an `IdentityHash` (`VesselSpawner.ComputeInventoryPayloadIdentityHash`, SHA-256 over a canonical STOREDPART walk with slot/quantity ignored and transient values stripped, module state in identity). Delivery writes via `LiveDeliveryWriters.WriteInventory` (store only). No remove-from-source inventory writer exists.

**Codec discipline: three surfaces, all sparse-additive + sentinel-gate + null-preserve + empty->null.** Recording-side proof I/O = `RouteProofCodec` (single chokepoint from both `ParsekScenario` OnSave/OnLoad and `RecordingTreeRecordCodec`); recording-side hash = `RouteProofHasher` (sentinel gate widened in M2 to `hasWindows || hasOrigin || hasRunManifest || hasHarvestWindows`); route-side = `RouteCodec` (sparse omit-at-default convention; `recordedDockUT` is an accepted unconditional-write wart, NOT the precedent to follow). Deep-clone forwarding rides THREE sites: `Recording.ApplyPersistenceArtifactsFrom`, `Recording.DeepClone`, and `ParsekFlight.ApplyCapturedLogisticsMetadataToRecording` (which intentionally does NOT forward connection-window/origin-proof/transfer fields - they ride `ApplyPersistenceArtifactsFrom` + `CreateMergeBranch`; do not double-write). `RouteSourceRef.RouteProofHash` gates `SourceChanged` on load.

## 3. Design decisions

**Open questions - DECIDED 2026-06-14 (maintainer sign-off recorded):** OQ1 -> RESEQUENCE multi-origin + escrow (19.2.5) to M4; M3 = pickup + mixed single-window (M3a). OQ2 -> window-local admit ALWAYS, full closure only with complete run manifests. OQ3 -> window-local inventory pickup, defer full-run inventory fields. OQ4 -> new `RouteCargoPickedUp = 29` + re-keyed replay backstop. Rationale for each below; section 5 (M3b) is now M4 scope.

- **OQ1 - Multi-origin scope: resequence to M4 (RECOMMENDED), or build the gate+map as M3 infrastructure?** True multi-origin (distinct depots A and B) requires N loaded windows, which requires lifting the `MultipleConnectionWindows` reject - the M4 multi-window acceptance. The single-window loop path debits AT the dock crossing, so there is no dispatch-to-debit gap to escrow until multi-window exists (19.7 item 6 confirms this). **Recommendation: M3 ships pickup + mixed single-window (M3a); the all-or-nothing source gate and the `RouteStore` cargo-escrow reservation map land with M4 multi-window, where they are first exercisable against real geometry.** This is a resequence, not a cut: the design doc's 19.2.5 work happens, just in M4 alongside the multi-window acceptance it depends on. Alternative if vetoed: build the gate + map now, exercised only by a synthetic 2-window fixture, with player-facing multi-origin still gated on M4 - more code shipped untested-in-game. The plan below is written for the recommended cut; section 5 lists what M3b adds if OQ1 keeps it in M3. The resequence ALIGNS with the design doc (M4's own bullet, design line 1642, already locates N-window acceptance in M4, and 19.2.5 line 1572 says the escrow map "first becomes real with multi-window runs (M3/M4)"); the only wrinkle is that the sentence "Multi-origin semantics (19.2.5) land here" is printed under M3's final bullet. If you bless the resequence, record it by editing the design 19.4 M3/M4 bullets in the SAME commit (move that sentence to M4) so the doc and this plan do not disagree on which milestone owns 19.2.5.

- **OQ2 - Closure degrade vs reject when run manifests are absent.** The full closure equation needs `launched` (anchor `StartTransportResources`) and `residual` (`EndTransportResources`) from a COMPLETE lineage of `RouteRunCargoManifest`s - the same presence gate M2 uses. A pickup/mixed run on a degraded lineage (ForceStop, BG transit, optimizer split, or any pre-M2 leg) has no launched/residual terms. **Recommendation: window-direction classification ALWAYS works from the four window corners (so the pickup is admitted and debited window-locally); the full-run closure VALIDATION (consumed/residual balance) runs only when the lineage carries complete run manifests, else it is skipped with a logged degrade - never silently falls back to the delivery-only path (which would DROP the pickup and re-introduce the assume-start-loaded blind spot).** This mirrors M2's degrade-to-legacy philosophy but must not degrade a pickup into invisibility. **Safety floor (review-confirmed):** the window-local pickup debit is `min(endpointLoss, transportGain)` from the resolved endpoint - self-contained per window, independent of the full-run terms - so it cannot over-debit a source even on a degraded lineage where `launched` / `residual` are unavailable. The window-local debit is the M3a correctness floor; full-run closure is an added validation when manifests are complete, never a precondition for admitting the witnessed pickup.

- **OQ3 - Inventory closure scope: window-local (RECOMMENDED for M3) or full-run.** Inventory is non-fungible and has no harvested provenance, so an unwitnessed transport inventory gain must fail closed. Full-run inventory closure needs new `Start/EndTransportInventory` fields on `RouteRunCargoManifest` (deferred from M2, `RouteProofMetadata.cs:160`) plus a recorder hook + codec + hasher + 3 clone sites. **Recommendation: M3 does inventory pickup WINDOW-LOCALLY (the window's witnessed inventory loaded term, debited from the source), and defers full-run inventory closure (consumed/residual across legs) until a real inventory-route playtest demands it.** Inventory-payload identity tests (bullet 5) are satisfied window-locally. If full-run inventory IS added later, note the deep-clone forwarding asymmetry (review correction): two of the three sites call `RouteRunCargoManifest.DeepClone()` wholesale (so a new field forwards automatically), but the END-half OVERWRITE branch in `ParsekFlight.ApplyCapturedLogisticsMetadataToRecording` copies INDIVIDUAL run-manifest fields (`EndTransportResources` + `EndCaptured`), NOT via `DeepClone`, so it needs an explicit new line for the inventory END field - "DeepClone at three sites" understates that one site.

- **OQ4 - Pickup ledger row: new `RouteCargoPickedUp` GameActionType (RECOMMENDED) or reuse `RouteCargoDebited`.** This is a CORRECTNESS issue, not cosmetics: the `EmitLoopCycle` ELS replay backstop keys on the `RouteCargoDelivered` row (`:981`), which a PICKUP-ONLY route never emits - so the idempotency guard never fires and the endpoint debit RE-APPLIES every save/reload. **Recommendation: add a `RouteCargoPickedUp` GameActionType (legible for the M6 per-cycle flow display) AND re-key the per-cycle replay backstop on a row every cycle emits regardless of direction (`RouteDispatched`, `(RouteId, RouteCycleId)`), so the guard is direction-agnostic.** Reusing `RouteCargoDebited` is cheaper but conflates dispatch-time origin debit with per-window pickup and still needs the replay re-key. **`RouteCargoPickedUp = 29`** (the GameActionType enum is non-contiguous: `RouteDispatched=23`, `RouteCargoDebited=24`, `RouteCargoDelivered=25`, `RouteRecoveryCredited=28` - so 29, not 26). The re-key is LOOP-PATH-ONLY: the second `IsDeliveryAlreadyInLedger` call in the legacy `ApplyDelivery` path stays on `RouteCargoDelivered` (it is dead for loop routes, M1 D11). **Full integration surface for the new type (review correction - more than the original list):** GameActionType enum value; `ParsekScenario` OnSave/OnLoad; **`LedgerLoadMigration.IsResourceImpactingAction` MUST get an explicit `case` returning false** (no funds; mirror `RouteCargoDelivered`) or the `IsResourceImpactingAction_Theory` forcing test fails; **`FundsModule` must route it as a non-funds row** (pickup emits zero funds, D6); **`SupersedeCommit.IsWorldStateChangingRecordingAction` MUST get an explicit `case ... RouteCargoPickedUp:` returning false** - a new type does NOT inherit the existing route-rows-return-false; with no case it falls through to `return true` and supersede would strict-block/retry on a pickup row; `RouteModule` handling (observe-only, mirror `ProcessCargoDebited`).

**Decisions:**

- **D1 - Capture is unchanged for resource pickup.** No recorder hook, no new `RouteConnectionWindow` field, no `RouteProofCapture` change. The four window corners already witness both directions. (Inventory pickup also reuses the existing four inventory corners - D7.)

- **D2 - Classify, do not reject; mirror the delivery builder with the sign flip and KEEP the undefined-name exclusion.** `HasResourcePickup` / `HasInventoryPickup` stop driving `MixedPickupDelivery` for an admissible window. New `BuildResourceLoadManifest` (mirror of `BuildResourceDeliveryManifest`): per routable name, `loaded = min(endpointLoss, transportGain)` when both > 1e-9, **excluding undefined resource names exactly as the delivery builder does** (`ResourceTransferability.IsRoutableResource`, M2 D2 parity). The REJECTION-direction reads (`HasResourcePickup` used as a presence detector) keep seeing undefined names so an uninstalled mod cannot flip a recorded pickup from rejected into a phantom admitted term (fail-closed). `MixedPickupDelivery = 4` stays a defined enum value (it becomes unreachable for admissible windows but must not be renumbered/removed - codec/save stability); it still fires for a window that is neither cleanly classifiable nor closeable.

- **D3 - The flow-closure equation, implemented.** Per resource (and per exact `InventoryPayloadItem` identity, window-locally per OQ3): `launched + Sum(loaded) + Sum(harvested) = Sum(delivered) + consumed + residual`. Terms: `launched` = anchor `RouteRunCargoManifest.StartTransportResources` (0 for docked/harvest start); `Sum(loaded)` = sum over pickup windows of the `loaded` term (D2); `Sum(harvested)` = the M2 `HarvestedManifest` (unchanged); `Sum(delivered)` = sum over delivery windows of the existing `delivered` term; `residual` = `EndTransportResources`; **`consumed := launched + Sum(loaded) + Sum(harvested) - Sum(delivered) - residual`** (the balancing slack). REJECT only when the pre-clamp slack is `< -GainEpsilon` (over-delivery / phantom cargo: more left than ever arrived) via a new `RouteAnalysisStatus.FlowDoesNotClose = 8` with a `RejectDetail` naming the resource and the unaccounted quantity (mirrors `UntrackedCargoGain`'s detail surface). A POSITIVE slack is legitimate burn/consumption, never a rejection. Structure `ComputeFlowClosure` to SUM over a window LIST (length 1 in M3) so M4 multi-window is a fill-in, not a rewrite (cheap, M4-ready). The closure VALIDATION is presence-gated per OQ2; window-direction admission is not. **Composition with the M2 gain check (review correction):** `RouteHarvestAnalysis` rejects `UntrackedCargoGain` when dock-time transport gain (`DockTransportResources - anchor StartTransportResources`) exceeds witnessed harvest - but a loaded (pickup) window LEGITIMATELY raises dock transport above anchor-start, so it would trip `UntrackedCargoGain` BEFORE closure runs. The harvest gain check must therefore treat each window's `loaded` term as a witnessed source (a third covering term alongside harvest windows + boundary bridges) so a pickup window does not false-reject as an untracked gain; closure (D4) then runs after.

- **D4 - Gate ORDER in `AnalyzeWindow` is preserved.** endpoint-proof -> (legacy) undocked-start -> [was: mixed-pickup reject] -> manifest build (now BOTH directions) -> (harvest) untracked-gain -> flow closure (D3) -> refined undocked-start -> Eligible. The mixed-pickup reject is REPLACED by classification, but workflow/undocked-start errors keep precedence over manifest-level closure complaints, and closure runs AFTER the harvest gain check engages (closure needs `HarvestedManifest`), exactly as M2 deferred the refined undocked-start gate to after the gain check. **Two gate fixes (review corrections):** (a) the `NoDeliveryManifest` reject (`RouteAnalysisEngine.cs:279-291`) fires when BOTH the resource and inventory DELIVERY manifests are empty - a pure-pickup window has no delivery manifest and would reject HERE before reaching Eligible, so widen the gate to "no delivery AND no load/pickup manifest"; (b) the `UntrackedCargoGain` composition note in D3 (the harvest gain check must see the loaded term as a witnessed source).

- **D5 - Reverse-direction physical writer: reuse M1 for resources, add the inventory remove writer.** Resource pickup REUSES `LiveOriginDebitWriters.WriteResourceDebit` re-aimed at the per-window pickup ENDPOINT vessel (resolved through `RouteEndpointResolver`, which resolves any `RouteEndpoint` by PID with a surface-proximity fallback; the window carries `EndpointAtDock` + `TransferTargetVesselPid`) sourcing amounts from the per-window load manifest instead of `route.CostManifest`. No sign flip inside the writer (it already removes); the flip is SCOPE (origin vessel -> per-window endpoint) + manifest source. **Planner refactor IN SCOPE (review correction):** the probe (`LiveOriginCargoProbe`) takes `(Vessel, isLoaded)` and the writer takes a prebuilt `OriginDebitPlan` - both compose unchanged - but `RouteOriginDebitPlanner.PrepareDebit(Route, IOriginCargoProbe)` reads `route.CostManifest` directly, and the `LiveOriginDebitWriters` ctor stores the `Route`. So a per-window manifest needs a planner signature change (accept an arbitrary `Dictionary<string,double>`, or a sibling pickup planner) and a ctor adjustment; this is real work, not a verbatim reuse. The "credit transport" half is BOOKKEEPING ONLY (transport never materializes, 19.2.3) - a ledger row, no physical write. Source availability REUSES `LiveOriginCargoProbe.ProbeResourceStored` pointed at the endpoint, planned by `RouteOriginDebitPlanner.PrepareDebit`. Loaded-gate (`vessel.loaded && !vessel.packed`) captured ONCE PER VESSEL (a two-direction applier touching multiple endpoint vessels captures it per vessel, never hoisted). Loop-path-only (M1 D11 parity); revert via rewind quicksave + ELS `(RouteId, RouteCycleId)` replay keys.

- **D6 - Two-direction applier at one dock crossing.** `EmitLoopCycle` generalizes its existing two-half shape (debit-half `EmitDispatchDebit` then credit-half `ApplyDelivery`) so one crossing fires, under one `cycleId`: the pickup half (endpoint debit + transport-credit row) AND the delivery half (endpoint credit). `RouteDispatched` stays Seq0 (the `RouteModule.ProcessCargoDelivered` out-of-order guard requires dispatch before delivery at the shared UT); pickup/debit/delivery rows get a deterministic Sequence assignment. **A pure-pickup route SKIPS the delivery half** (emits no `RouteCargoDelivered`) - that decision is precisely what makes the OQ4 replay hole real AND the re-key necessary (today `EmitLoopCycle` ALWAYS calls `ApplyDelivery`, which emits a `RouteCargoDelivered` row even on an empty plan, so without the skip there would be no hole). The per-cycle replay backstop is re-keyed (OQ4) so a pickup-only route is idempotent across reload, and the re-keyed guard MUST preserve the existing `route.CompletedCycles` bump in the replay branch (`EmitLoopCycle:983-988`) or a pickup-only route replay-skips forever. Pickup windows emit ZERO funds (loaded-en-route cargo debits its physical source, never funds, 19.2.4); the funds charge keeps the M2 launch-manifest/stop-snapshot basis untouched.

- **D7 - Inventory pickup (window-local, OQ3).** Classify (mirror `BuildInventoryDeliveryManifest` with the sign flip, identity carried intact); source availability = a new find-slot-by-`IdentityHash` probe (mirror of `LiveDeliveryCapacityProbe.ProbeFirstEmptyInventorySlot`); the NEW inventory-remove writer (the only genuinely new mutation in M3): debit the source STOREDPART slot (loaded `ClearPartAtSlot`, unloaded remove the matching-hash STOREDPART node - inverse of `LiveDeliveryWriters.WriteInventoryUnloaded`'s add) and credit the transport, matched strictly by `IdentityHash`. This relates to the M1 `inventory-origin-debit-unsupported` carve-out (`LiveRouteRuntimeEnvironment.cs:127-136`, `RouteOriginCargoCheck.RequiresInventoryDebit`) - **but that hold gates on `route.InventoryCostManifest` (the dispatch-time ORIGIN inventory cost on a non-KSC origin), which is EMPTY for an originless pure-pickup run (D8), so it never fires for the pickup path D7 actually builds** (review correction). The per-window pickup inventory writer is the primary M3 deliverable; lifting the origin-dispatch hold additionally requires wiring the same remove-writer to the origin-dispatch path. Decide whether the new inventory-remove writer serves BOTH paths or only the per-window path; if both, lift the hold only AFTER it lands (else inventory origin routes dispatch and duplicate matter). Multiple stored parts of one `IdentityHash` with a partial load need a deterministic match rule (lowest slot index, replay-stable). Multiple stored parts of one `IdentityHash` with a partial load need a deterministic match rule (lowest slot index, replay-stable).

- **D8 - Route/RouteStop shape: a per-stop pickup manifest, sparse codec, no new window field.** `RouteStop` gains `PickupManifest` (`Dictionary<string,double>`) + `InventoryPickupManifest` (mirror of `DeliveryManifest` / `InventoryDeliveryManifest`), serialized as new sparse `PICKUP_MANIFEST` / `INVENTORY_PICKUP_MANIFEST` nodes (verbatim copies of the delivery-manifest codec call sites with new node names, omit-when-empty). `RouteAnalysisResult` gains the parallel `ResourceLoadManifest` / `InventoryLoadManifest`. The per-window direction classification is DERIVED from the four window corners at analysis time, NOT stored on `RouteConnectionWindow` and NOT hashed - this keeps pre-M3 proof-hash byte-identity automatic (D9). `RouteBuilder` admits an originless pickup run (the endpoint is the source, not `route.Origin`) without the v0 `endpoint-missing` reject, mirroring how M2's harvest-origin branch was added.

## 6. Risk register

- **Replay-key hole (CORRECTNESS, OQ4).** Pickup-only routes emit no `RouteCargoDelivered`, so today's replay backstop never fires and the endpoint debit re-applies every reload. Fix BEFORE the applier ships (Phase 4).
- **Hash byte-identity (M1/M2 lesson).** Any unconditional new serialized field flips every pre-M3 hash and marks all routes `SourceChanged`. Derive the pickup direction; keep pickup is no new window field; sparse everything; pin `Hash_PreM3Recording_ByteStable`.
- **Fail-open undefined names.** The load-ADMISSION builder must exclude undefined resource names (D2); the rejection-direction read must keep seeing them. Splitting classify-vs-reject must not let a mod uninstall flip a recorded pickup from rejected to admitted.
- **consumed sign (D3).** `consumed` is the balancing slack; reject only on negative slack. Getting it wrong turns every fuel-burning run into a closure rejection.
- **Presence-gate degrade dropping the pickup (OQ2).** A degraded lineage must classify the pickup window-locally, never silently fall back to the delivery-only path.
- **Loaded-gate per-vessel capture.** A two-direction applier touching multiple endpoint vessels captures `loaded && !packed` once per vessel; never hoist one flag across vessels.
- **Inventory identity cross-vessel (D7).** `ComputeInventoryPayloadIdentityHash` must hash identically for the depot STOREDPART and the witnessed item; any unstripped depot-side value splits identity and fails closed. The existing live-move test is same-vessel only - add a cross-vessel + proto-vs-loaded test.
- **RouteModule out-of-order guard.** `RouteDispatched` stays Seq0; pickup/debit/delivery rows get deterministic Sequence so the ledger walker sees dispatch before delivery at the shared UT.
- **Revert safety for the endpoint debit.** The new endpoint (resource + inventory) debit writes a third-party vessel's tanks/inventory, same as M1 origin debit - reversible only if the pickup source vessel is in the rewind quicksave; loop-path-only.
- **M4 leak.** Keep `RouteStop` length 1, `RecordedDockUT` single, `RouteLoopClock` single-dock, `MultipleConnectionWindows` reject in place. A second window/stop in M3 crosses into M4.
