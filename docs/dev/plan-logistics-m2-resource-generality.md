## 1. Summary + scope

M2 makes the logistics pipeline officially resource-agnostic (any `PartResourceDefinition`-backed resource routes, modded included) and adds the Harvested provenance kind so mining routes (scenario family 4, design line 1583) work end to end. Five deliverables:

1. **Transferability rule, stated in one place:** any defined resource is routable; ElectricCharge/IntakeAir stay excluded; an undefined resource name in a snapshot is excluded from analysis manifests and logged; zero-cost defined resources route normally.
2. **CRP-style mod-resource fixtures** across manifest capture, analysis, delivery/debit, funds cost - xUnit, via the existing injected-lookup seams plus one new definition-exists seam.
3. **Harvest provenance:** (a) full-run transport-scoped start/end manifests for ALL recordings (new `RouteRunCargoManifest`), (b) harvest-window capture pinned to `BaseConverter`-derived module activity (covers `ModuleResourceHarvester`, `ModuleResourceConverter`, `ModuleAsteroidDrill`, `ModuleCometDrill`, and modded derivatives by base class, not by list), (c) analysis admits witnessed harvested gains and fail-closes on untracked gains with a player-language rejection naming the unaccounted quantity, (d) all capture additive; recordings immutable; old recordings analyze exactly as today.
4. **Funds-charge basis decision** (19.2.4 deferred note): recommendation below (D9 / OQ1).
5. **Background-production non-goal paragraph** in the design doc.

**The M3 boundary, drawn explicitly.** M2 implements only the GAIN side of flow closure: every positive between-window transport gain must be covered by witnessed harvest windows, else the recording is rejected naming the quantity. M2 does NOT compute or check `consumed`/`residual` (the full 19.2.4 equation), does NOT replace assume-start-loaded delivery attribution, does NOT admit pickup direction (`HasResourcePickup`/`MixedPickupDelivery` unchanged - which also means harvest gains INSIDE a docked window, i.e. drilling while docked at the destination, still reject; documented limitation lifted by M3 flow accounting), does NOT do inventory-payload closure (drills produce resources only; inventory gains stay unaccounted as today), and does NOT support N>1 windows (M4). Harvest capture on background-recorded legs (player switches away from a loaded drilling transport) is also deferred: a leg that transits background VOIDS its run manifest (review finding 2), so the lineage lacks complete manifests and the gain check DEGRADES TO LEGACY analysis for that tree - the stated fail mode is "absent manifest -> legacy", not a named-quantity rejection. The workflow rule is "stay with the mining vessel" (mirrors the M1 undocked-start workflow-rule philosophy). Similarly, run manifests complete only on ACTIVE stops (`BuildCaptureRecording` paths); `ForceStop` scene-change stops never complete one and degrade the same way (review finding 4) - the docs say "active-stop recordings", never "all recordings".

## 3. Design decisions

**Open question for the maintainer (one):**

- **OQ1 - funds-charge basis.** Recommendation (ADOPTED for this implementation; flagged in the PR for sign-off): **key the RESOURCE term of `ComputeDispatchFundsCost` on the full-run START transport manifest** (the new `RouteRunCargoManifest.StartTransportResources` of `SourceRefs[0]`'s recording) **when present, falling back to today's stop-snapshot walk when absent; parts term unchanged** (stop snapshot). Rationale: (i) it is the 19.2.4 "restated cleanly on the launched term" option - the player pays for what KSC supplied at launch, (ii) it stops charging funds for harvested cargo aboard at dock (which after M2 would directly violate "Harvested - Debit: none", 19.2.2 item 3), (iii) it fixes the known under-count of burned transit fuel (19.2.4 line 1564), and (iv) the fallback contains the change to routes built from NEW recordings - every existing route keeps its exact cost. The genuinely contestable part: new KSC routes get slightly MORE expensive (launch-tank contents, including fuel burned in transit, are now charged) while mining-launch routes get cheaper (ore no longer charged); and the parts term deliberately stays on the surviving part set (boosters still uncharged) because no costing-grade start part list is recorded and charging the full launch stack is a bigger economics change than M2 needs. Alternative if vetoed: keep dock-arrival contents but subtract witnessed harvested amounts per resource - smaller player-visible delta, weirder contract, same new-data dependency.

**Decisions:**

- **D1 - transferability rule gets one authority.** New pure `Logistics/ResourceTransferability.cs`: `IsRoutableResource(string name, out string excludeReason)` = not-EC/IntakeAir AND defined; `internal static Func<string, bool> DefinitionLookupOverrideForTesting` seam; production lookup via `PartResourceLibrary.Instance.GetDefinition != null`, and a null library (xUnit, early load) treats names as defined with a one-shot Verbose - never reject everything headlessly. `RouteAnalysisEngine.IsIgnoredResource` keeps the EC/IntakeAir core but the manifest walks add the defined check. The class doc comment IS the stated rule (work item 1), quoted in the design-doc flip.
- **D2 - exclusion applies at ANALYSIS, capture stays permissive, and it is DIRECTION-SENSITIVE (review finding 8).** `VesselSpawner.ExtractResourceManifest` keeps recording whatever names the snapshot carries (recordings are immutable witnesses; a definition check at capture would bake mod-install state into recorded data). Undefined names are excluded ONLY from ADMISSION-direction outputs: the delivery manifest, `CostManifest`, and the funds charge. They stay VISIBLE to REJECTION-direction checks: `HasResourcePickup` keeps rejecting `MixedPickupDelivery` on an undefined-name pickup (excluding it there would let a mod uninstall flip a reject into Eligible - fail-open), and an undefined-name positive gain in the new gain check counts as UNACCOUNTED (fails closed). Each admission-side skip is logged (rate-limited in the 1 Hz sweep). Consequence: uninstalling a resource mod degrades a recording to `NoDeliveryManifest` (or keeps its rejection) instead of routing a phantom resource; reinstalling restores it. The live writers need no change (they only touch tanks that exist).
- **D3 - run-manifest shape: new `RouteRunCargoManifest`, presence-gated, with a WRITE-ONCE lifecycle (review findings 2, 4, 13).** Fields: `TransportPartPersistentIds` (scope set = `VesselSpawner.CollectPartPersistentIds(lastGoodVesselSnapshot)` at start, identical to the start-docked proof's scope, `RouteProofCapture.cs:208`), `StartTransportResources`, `EndTransportResources` (extraction scoped to the start pid set, so parts decoupled mid-run drop out of the END manifest - losses, which M2 does not check). The inventory halves (`Start/EndTransportInventory`) are DEFERRED TO M3 (review finding 13: nothing in M2 reads them, and InventoryPayloadItem STOREDPART snapshots inflate persistent.sfs for every recording; the sparse codec makes adding them later free). Why not reuse `StartResources`/`EndResources`: wrong scope, optimizer-degraded, and - decisive - the spec requires old recordings to analyze AS TODAY; a new node's absence is the clean old/new discriminator.
  **Lifecycle rule (mandatory; round-2 correction 1 - the discriminator is RECORDING BIRTH, not the recorder-start flavor):** `isPromotion: true` is ALSO used for split children (`ParsekFlight.cs:4846`), merge children (`:5650`), and chain-boundary continuations (`:12934`) - new Recordings that MUST capture, or the D6 presence gate is unsatisfiable on every M2 scenario (the undock-split anchor and the dock-merge window leg would never carry manifests). So:
  1. The START half is captured iff the tree's active Recording is at its BIRTH (no prior samples) AND has no start half yet (the write-once guard). Capturing populations: root/user start, undock-split child, dock-merge child, chain-segment birth. Never-capture populations: BG-promotion of an EXISTING recording, quickload resume (`PrepareQuickloadResumeStateIfNeeded`, `FlightRecorder.cs:6110`) - a re-captured mid-run baseline would fold prior gains into "start cargo" and bypass the gain check (review finding 2). Written onto the tree recording IMMEDIATELY (write-once), not held solely in recorder pending state. Implementation footnote (round-2 nit 8): `InsertBoundaryAnchorAndSnapshot(Vessel v)` does not currently see the promotion flag - thread it (or the birth predicate) from `StartRecording` scope (`FlightRecorder.cs:6076`).
  2. The START half is write-once; the END half is OVERWRITE-PER-ACTIVE-STOP (round-2 correction 6): a chain-boundary stop abandoned by `ResumeAfterFalseAlarm` has already completed an END half, and the eventual real stop must replace it or post-resume drilling double-counts. Harvest windows live on the RECORDER-side list until stop forwarding; the close-at-stop mutation is what the false-alarm reopen unwinds.
  3. A recording that transits BACKGROUND (`TransitionToBackground` - the BG flush runs with `CaptureAtStop == null`, `FlightRecorder.cs:7740-7757` / `ParsekFlight.cs:12369-12396`) VOIDS its run manifest (cleared on the tree recording, Warn-logged): its END can never be captured trustworthily, and a void manifest makes the D6 presence gate degrade that tree to legacy analysis.
  4. The END half completes only on ACTIVE stops (`BuildCaptureRecording` paths: StopRecording, chain boundary, pid change); `ForceStop` leaves it absent. **The D6 presence gate requires a COMPLETE manifest - both halves non-null** (round-2 correction 5): a start-only ForceStop leg degrades that tree to legacy, never feeds a null END into the bridge math. Completion extracts from `capture.VesselSnapshot` / `lastGoodVesselSnapshot` scoped to the START pid set, mirroring `AttachEndManifestsAndForwardToCapture` (`RouteProofCapture.cs:314-317`) - NEVER from a live `v.parts` walk at the stop frame (review finding 5: at the dock pid-change stop the live vessel is the merged stack and same-frame crossfeed equalization deflates values; `pendingDockSelfSnapshot` exists for exactly this reason, `ParsekFlight.cs:5496-5502`).
- **D4 - harvest capture shape: HARVEST WINDOWS on converter-activity threshold crossings.** Chosen over per-window bracketing because it matches the repo principle directly: record threshold crossings (converter set becomes active / becomes idle - player actions, inherently debounced), snapshot coarse state only at the crossings, store nothing per-frame. `RouteHarvestWindow` DTO: `WindowId`, `StartUT`, `EndUT` (NaN while open), `OpenedAtRecordingStart`, `ClosedAtRecordingStop`, `StartTransportResources`, `EndTransportResources` (live-part walks, EC/IntakeAir excluded, scoped to the transport's own parts), `ActiveConverters` (list of `partPid:moduleClass:ConverterName` strings, diagnostic), plus the open-time location (`BodyName`, `Latitude`, `Longitude`, `Altitude`, `SituationAtOpen`) for the D7 harvest-origin endpoint. Activity = `BaseConverter.IsActivated` on a converter list cached at recording start (re-scanned when `v.parts.Count` changes, so EVA-constructed drills are caught; Unity-null entries count as inactive, so a staged-away drill closes the window). Keying on the `BaseConverter` base class (the `TimeJumpManager.cs:804` precedent) covers `ModuleResourceHarvester`, `ModuleResourceConverter`, both asteroid/comet drills, and modded derivatives - satisfying the M-MIS-11 cross-note with no module list. Harvested manifest of a window = per-resource positive delta (end minus start); an activated-but-stalled drill nets 0 harmlessly (this also makes `IsActivated`-while-not-actually-harvesting - no ground contact, no asteroid - harmless: the window exists, the delta is 0). **Warp rule (review finding 10):** `OnPhysicsFrame` early-returns on `isOnRails` (`FlightRecorder.cs:7806`) while loaded converters keep producing at warp, so the poll alone would miss warp-period production. The capture therefore re-baselines at the RAILS TRANSITIONS: on rails-entry with a window open, the window stays open (production continues inside it - correct); on rails-entry with converters active but no window open (activation raced the poll), open a window AT rails-entry; on rails-exit, run one immediate poll so a toggle that happened during warp opens/closes its window at the exit boundary. Hook: rails-ENTRY via the recorder's existing `OnVesselGoOnRails`-adjacent seam (`FlightRecorder.cs:9877` region); rails-EXIT is NOT one seam (`isOnRails = false` at ~5 sites) - funnel it through a pending flag checked at the top of the next `OnPhysicsFrame` instead of hooking per-site (round-2 nit 8). **Stop/false-alarm rule (review finding 4):** the close-at-stop mutation must be UNWINDABLE - `ResumeAfterFalseAlarm` (`FlightRecorder.cs:7411-7486`) discards a chain-boundary `CaptureAtStop` and resumes the same recording; mirroring the #287 `RemoveLastEmittedTerminals` precedent (:7439), a window closed by the abandoned stop is REOPENED (clear `EndUT`/`ClosedAtRecordingStop`/end manifest) so subsequent drilling stays accounted. **Per-frame cost budget, stated:** N boolean property reads per physics frame on the active vessel only (N = cached converter count, typically 0-8; zero allocations, zero per-part scans); one O(parts) manifest extraction per window open/close (player-toggle or rails-transition frequency) and per recording start/stop; an int part-count compare per frame for the cache guard.
- **D5 - the catch-up trap, resolved by open-at-start windows + boundary bridging.** (i) If a converter is already active at recording start, a window opens AT start with the start manifest - so the stock `lastUpdateTime` catch-up burst that fires in the first frames after a vessel loads/unpacks lands INSIDE an open window and attributes as harvested. (ii) Recording stop closes open windows (`ClosedAtRecordingStop = true`); the next chain segment reopens (`OpenedAtRecordingStart = true`). The gap BETWEEN two legs (where a burst lands if it beats the start snapshot - frame timing that cannot be promised from code reading) is handled in analysis by the **boundary-bridging rule**, SCOPED (review BLOCKER 1): a positive boundary delta (leg N+1 start manifest minus leg N end manifest) counts as harvested iff ALL of: (a) the seam lies STRICTLY BEFORE the window's `DockUT` - the dock-merged child's start manifest is NEVER a bridge operand (it is scoped to the COMBINED transport+depot stack, `ParsekFlight.cs:5473-5476`, so bridging across the dock seam would credit the depot's entire inventory as harvested - the false-admit + zero-debit exploit the review caught); (b) both legs' `TransportPartPersistentIds` sets MATCH (same-scope manifests; merge/board seams with differing scopes are unbridgeable and any positive delta there is unaccounted, fails closed); (c) leg N's last window is `ClosedAtRecordingStop` or leg N+1's first window is `OpenedAtRecordingStart` (a converter was witnessed active at the boundary). This makes the analysis correct regardless of which side of the start snapshot the burst lands on, without ever bridging across a vessel-composition change. (iii) Parsek time jumps: `TimeJumpManager.FixResourceConverterTimestamps` already zeroes the burst on every loaded vessel, so a window spanning a Parsek jump records zero gain - intentional (a jump is not simulated production time), documented in the design doc, and it means the recorder needs NO jump-awareness of its own. (iv) An in-game investigation test (Phase 6) verifies the live ordering of load-time catch-up vs the start snapshot and pins whichever branch fires.
- **D6 - analysis admission rule (the M2 slice of flow closure).** New pure entry `RouteHarvestAnalysis.CheckTransportGains(...)` called from `AnalyzeWindow` after the manifest build, ENGAGED ONLY when every recording on the transport lineage carries a `RouteRunCargoManifest` (else legacy behavior, logged once per analysis). Mechanics: `CollectTransportLineage(tree, sourceRec, window)` walks parents from the window's source recording via `ParentBranchPointId`/`BranchPoint.ParentRecordingIds`, disambiguating multi-parent merge points by part-pid overlap with `window.TransportPartPersistentIds` (the depot's BG leg never enters); ambiguity falls back to legacy with a log. The GAIN ANCHOR is NOT the tree root and is found by a BACKWARD WALK (round-2 correction 2 - a first-by-UT superset rule would select the combined-stack root on the depot run and reinstate the inert check): start at the window's own leg and extend the anchor EARLIER across a seam only while the earlier leg's run-manifest pid scope MATCHES the window's `TransportPartPersistentIds`, or is a superset AND the seam is a chain/continuation boundary - NEVER across an Undock/Dock/Board branch point (`BranchPointType` is on the branch point, available to the walker). In practice: depot run anchors at the undock-split child (the walk stops at the Undock BP); KSC launch anchors at the root (chain seams, scope superset covers jettisoned boosters, and `max(0, ...)` clamps booster-fuel losses). Pre-anchor gains (drilling while docked at the origin) are OUTSIDE the check until M3, stated as a documented limitation. **Span limit (round-2 correction 2b): `harvested[r]` sums only windows with `StartUT >= anchorLeg.StartUT` and `<= window.DockUT`** - pre-anchor windows must not inflate the harvested total (fail-open mask). PersistentIds are craft-baked, not launch-unique (CLAUDE.md), which is exactly why scope ambiguity falls back to legacy with a log, never picks-larger-overlap. The rule per defined resource: `fullRunGain[r] = max(0, window.DockTransportResources[r] - anchorLeg.RunManifest.StartTransportResources[r])`; `harvested[r]` = sum of window positive deltas + bridged boundary deltas (D5, anchor-to-dock seams only); undefined-name positive gains count as unaccounted (D2); if `fullRunGain[r] - harvested[r] > epsilon` reject with **new `RouteAnalysisStatus.UntrackedCargoGain = 7`** and a detail string naming resource and quantity. Detail threading: `RouteAnalysisResult.RejectDetail` + a `FormatRejectMessage(status, detail)` overload (single-arg overload kept); `RouteNearMiss` gains a `RejectDetail` field populated in `DeriveNearMisses` (`RouteCandidateFinder.cs:248-253`) and `DescribeNearMiss` passes it through (review finding 12 - without this hop the near-miss list shows no quantity). Gate ORDER (review finding 11): the refined undocked-start verdict needs harvested totals, which need the manifest build - so when run manifests are present the undocked-start check is DEFERRED until after the gain check (two-phase gate); on the legacy path (no manifests) it keeps today's position (:184-197). This reorders rejection precedence for harvest-data recordings only; fixture churn is expected and called out in risk 10. The undocked-start gate is refined, not removed: undocked start + non-empty inventory delivery, or any delivered resource with `delivered[r] > harvested[r] + epsilon`, still rejects `UndockedStartOrigin`; undocked start whose delivered resources are FULLY harvest-covered becomes Eligible with `RouteAnalysisResult.HarvestedManifest` populated and a new `RouteAnalysisResult.IsHarvestOrigin = true`. Post-undock-tail gains (drilling after delivery) are outside the checked span by construction (the check ends at the window's dock).
- **D7 - harvest-origin routes (minimal route-side shape).** `Route.IsHarvestOrigin` (bool, sparse codec key `isHarvestOrigin`); `RouteBuilder` gains a third origin branch building a display-only endpoint from the FIRST harvest window's open location (pid 0, `IsSurface` from `SituationAtOpen`), `CostManifest` empty (Harvested debits nothing, 19.2.2 item 3), delivery manifests on the stop unchanged.
