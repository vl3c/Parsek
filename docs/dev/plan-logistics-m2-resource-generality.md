**Branch:** `logistics-m2-resource-generality` (worktree `Parsek-logistics-m2-resources`, from origin/main tip 9cee0b9cd, post-M1 / post-M6-hold-reasons / post-M-MIS-9-freeze)

## 1. Summary + scope

M2 makes the logistics pipeline officially resource-agnostic (any `PartResourceDefinition`-backed resource routes, modded included) and adds the Harvested provenance kind so mining routes (scenario family 4, design line 1583) work end to end. Five deliverables:

1. **Transferability rule, stated in one place:** any defined resource is routable; ElectricCharge/IntakeAir stay excluded; an undefined resource name in a snapshot is excluded from analysis manifests and logged; zero-cost defined resources route normally.
2. **CRP-style mod-resource fixtures** across manifest capture, analysis, delivery/debit, funds cost - xUnit, via the existing injected-lookup seams plus one new definition-exists seam.
3. **Harvest provenance:** (a) full-run transport-scoped start/end manifests for ALL recordings (new `RouteRunCargoManifest`), (b) harvest-window capture pinned to `BaseConverter`-derived module activity (covers `ModuleResourceHarvester`, `ModuleResourceConverter`, `ModuleAsteroidDrill`, `ModuleCometDrill`, and modded derivatives by base class, not by list), (c) analysis admits witnessed harvested gains and fail-closes on untracked gains with a player-language rejection naming the unaccounted quantity, (d) all capture additive; recordings immutable; old recordings analyze exactly as today.
4. **Funds-charge basis decision** (19.2.4 deferred note): recommendation below (D9 / OQ1).
5. **Background-production non-goal paragraph** in the design doc.

**The M3 boundary, drawn explicitly.** M2 implements only the GAIN side of flow closure: every positive between-window transport gain must be covered by witnessed harvest windows, else the recording is rejected naming the quantity. M2 does NOT compute or check `consumed`/`residual` (the full 19.2.4 equation), does NOT replace assume-start-loaded delivery attribution, does NOT admit pickup direction (`HasResourcePickup`/`MixedPickupDelivery` unchanged - which also means harvest gains INSIDE a docked window, i.e. drilling while docked at the destination, still reject; documented limitation lifted by M3 flow accounting), does NOT do inventory-payload closure (drills produce resources only; inventory gains stay unaccounted as today), and does NOT support N>1 windows (M4). Harvest capture on background-recorded legs (player switches away from a loaded drilling transport) is also deferred: such gains fail closed with the named quantity, and the workflow rule is "stay with the mining vessel" (mirrors the M1 undocked-start workflow-rule philosophy).

## 3. Design decisions

**Open question for the maintainer (one):**

- **OQ1 - funds-charge basis.** Recommendation (ADOPTED for this implementation; flagged in the PR for sign-off): **key the RESOURCE term of `ComputeDispatchFundsCost` on the full-run START transport manifest** (the new `RouteRunCargoManifest.StartTransportResources` of `SourceRefs[0]`'s recording) **when present, falling back to today's stop-snapshot walk when absent; parts term unchanged** (stop snapshot). Rationale: (i) it is the 19.2.4 "restated cleanly on the launched term" option - the player pays for what KSC supplied at launch, (ii) it stops charging funds for harvested cargo aboard at dock (which after M2 would directly violate "Harvested - Debit: none", 19.2.2 item 3), (iii) it fixes the known under-count of burned transit fuel (19.2.4 line 1564), and (iv) the fallback contains the change to routes built from NEW recordings - every existing route keeps its exact cost. The genuinely contestable part: new KSC routes get slightly MORE expensive (launch-tank contents, including fuel burned in transit, are now charged) while mining-launch routes get cheaper (ore no longer charged); and the parts term deliberately stays on the surviving part set (boosters still uncharged) because no costing-grade start part list is recorded and charging the full launch stack is a bigger economics change than M2 needs. Alternative if vetoed: keep dock-arrival contents but subtract witnessed harvested amounts per resource - smaller player-visible delta, weirder contract, same new-data dependency.

**Decisions:**

- **D1 - transferability rule gets one authority.** New pure `Logistics/ResourceTransferability.cs`: `IsRoutableResource(string name, out string excludeReason)` = not-EC/IntakeAir AND defined; `internal static Func<string, bool> DefinitionLookupOverrideForTesting` seam; production lookup via `PartResourceLibrary.Instance.GetDefinition != null`, and a null library (xUnit, early load) treats names as defined with a one-shot Verbose - never reject everything headlessly. `RouteAnalysisEngine.IsIgnoredResource` keeps the EC/IntakeAir core but the manifest walks add the defined check. The class doc comment IS the stated rule (work item 1), quoted in the design-doc flip.
- **D2 - exclusion applies at ANALYSIS, capture stays permissive.** `VesselSpawner.ExtractResourceManifest` keeps recording whatever names the snapshot carries (recordings are immutable witnesses; a definition check at capture would bake mod-install state into recorded data). `BuildResourceDeliveryManifest`, `HasResourcePickup`'s key walk, and the new gain/harvest math skip undefined names, each skip logged (rate-limited in the 1 Hz sweep). Consequence: uninstalling a resource mod degrades a recording to `NoDeliveryManifest` instead of routing a phantom resource; reinstalling restores it. The live writers need no change (they only touch tanks that exist).
- **D3 - run-manifest shape: new `RouteRunCargoManifest`, presence-gated.** Fields: `TransportPartPersistentIds` (scope set = `VesselSpawner.CollectPartPersistentIds(lastGoodVesselSnapshot)` at start, identical to the start-docked proof's scope, `RouteProofCapture.cs:208`), `StartTransportResources`, `EndTransportResources` (extraction scoped to the start pid set, so parts decoupled mid-run drop out of the END manifest - losses, which M2 does not check), `StartTransportInventory`, `EndTransportInventory` (`InventoryPayloadItem`-typed, reusing the proof codec helpers). Captured for every recording (gloops excluded, same guard as the origin proof). Why not reuse `StartResources`/`EndResources`: wrong scope, wrong inventory type, optimizer-degraded, and - decisive - the spec requires old recordings to analyze AS TODAY; a new node's absence is the clean old/new discriminator, where the legacy fields' presence is not.
- **D4 - harvest capture shape: HARVEST WINDOWS on converter-activity threshold crossings.** Chosen over per-window bracketing because it matches the repo principle directly: record threshold crossings (converter set becomes active / becomes idle - player actions, inherently debounced), snapshot coarse state only at the crossings, store nothing per-frame. `RouteHarvestWindow` DTO: `WindowId`, `StartUT`, `EndUT` (NaN while open), `OpenedAtRecordingStart`, `ClosedAtRecordingStop`, `StartTransportResources`, `EndTransportResources` (live-part walks, EC/IntakeAir excluded, scoped to the transport's own parts), `ActiveConverters` (list of `partPid:moduleClass:ConverterName` strings, diagnostic), plus the open-time location (`BodyName`, `Latitude`, `Longitude`, `Altitude`, `SituationAtOpen`) for the D7 harvest-origin endpoint. Activity = `BaseConverter.IsActivated` on a converter list cached at recording start (re-scanned when `v.parts.Count` changes, so EVA-constructed drills are caught; Unity-null entries count as inactive, so a staged-away drill closes the window). Keying on the `BaseConverter` base class (the `TimeJumpManager.cs:804` precedent) covers `ModuleResourceHarvester`, `ModuleResourceConverter`, both asteroid/comet drills, and modded derivatives - satisfying the M-MIS-11 cross-note with no module list. Harvested manifest of a window = per-resource positive delta (end minus start); an activated-but-stalled drill nets 0 harmlessly. **Per-frame cost budget, stated:** N boolean property reads per physics frame on the active vessel only (N = cached converter count, typically 0-8; zero allocations, zero per-part scans); one O(parts) live resource walk per window open/close (player-toggle frequency) and per recording start/stop; an int part-count compare per frame for the cache guard.
- **D5 - the catch-up trap, resolved by open-at-start windows + boundary bridging.** (i) If a converter is already active at recording start, a window opens AT start with the start manifest - so the stock `lastUpdateTime` catch-up burst that fires in the first frames after a vessel loads/unpacks lands INSIDE an open window and attributes as harvested. (ii) Recording stop closes open windows (`ClosedAtRecordingStop = true`); the next chain segment reopens (`OpenedAtRecordingStart = true`). The gap BETWEEN two legs (where a burst lands if it beats the start snapshot - frame timing that cannot be promised from code reading) is handled in analysis by the **boundary-bridging rule**: a positive boundary delta (leg N+1 start manifest minus leg N end manifest) counts as harvested iff leg N's last window is `ClosedAtRecordingStop` or leg N+1's first window is `OpenedAtRecordingStart` - i.e., a converter was witnessed active at the boundary; otherwise it is unaccounted and fails closed. This makes the analysis correct regardless of which side of the start snapshot the burst lands on. (iii) Parsek time jumps: `TimeJumpManager.FixResourceConverterTimestamps` already zeroes the burst on every loaded vessel, so a window spanning a Parsek jump records zero gain - intentional (a jump is not simulated production time), documented in the design doc, and it means the recorder needs NO jump-awareness of its own. (iv) An in-game investigation test (Phase 6) verifies the live ordering of load-time catch-up vs the start snapshot and pins whichever branch fires.
- **D6 - analysis admission rule (the M2 slice of flow closure).** New pure entry `RouteHarvestAnalysis.CheckTransportGains(...)` called from `AnalyzeWindow` after the manifest build, ENGAGED ONLY when every recording on the transport lineage carries a `RouteRunCargoManifest` (else legacy behavior, logged once per analysis). Mechanics: `CollectTransportLineage(tree, sourceRec, window)` walks parents from the window's source recording via `ParentBranchPointId`/`BranchPoint.ParentRecordingIds`, disambiguating multi-parent merge points by part-pid overlap with `window.TransportPartPersistentIds` (the depot's BG leg never enters); ambiguity falls back to legacy with a log. Ordered by start UT, the rule per defined resource: `fullRunGain[r] = max(0, window.DockTransportResources[r] - lineageRoot.RunManifest.StartTransportResources[r])`; `harvested[r]` = sum of window positive deltas + bridged boundary deltas (D5); if `fullRunGain[r] - harvested[r] > epsilon` reject with **new `RouteAnalysisStatus.UntrackedCargoGain = 7`** and a detail string naming resource and quantity. Detail threading: `RouteAnalysisResult.RejectDetail` + a `FormatRejectMessage(status, detail)` overload (single-arg overload kept); `DescribeNearMiss` passes the detail through. The undocked-start gate is refined, not removed: undocked start + non-empty inventory delivery, or any delivered resource with `delivered[r] > harvested[r] + epsilon`, still rejects `UndockedStartOrigin`; undocked start whose delivered resources are FULLY harvest-covered becomes Eligible with `RouteAnalysisResult.HarvestedManifest` populated and a new `RouteAnalysisResult.IsHarvestOrigin = true`. Post-undock-tail gains (drilling after delivery) are outside the checked span by construction (the check ends at the window's dock).
- **D7 - harvest-origin routes (minimal route-side shape).** `Route.IsHarvestOrigin` (bool, sparse codec key `isHarvestOrigin`); `RouteBuilder` gains a third origin branch building a display-only endpoint from the FIRST harvest window's open location (pid 0, `IsSurface` from `SituationAtOpen`), `CostManifest` empty (Harvested debits nothing, 19.2.2 item 3), delivery manifests on the stop unchanged. Dispatch: `RouteDispatchEvaluator.CheckEligibility` skips origin endpoint resolution for harvest origins (nothing to resolve, pid 0); `LiveRouteRuntimeEnvironment.OriginHasCargo` returns true with a rate-limited breadcrumb; `EmitDispatchDebit` keeps emitting the `RouteDispatched`/`RouteCargoDebited` pair for row-shape stability (empty manifest, zero funds, no physical write - the empty `CostManifest` makes the existing non-KSC path a structural no-op). `RouteCreationFormatters` origin label: "harvested en route".
- **D8 - mixed docked-origin + harvest runs: debit basis reduced by witnessed harvested.** `RouteBuilder` sets `CostManifest[r] = max(0, deliveryManifest[r] - harvested[r])` when harvest data is present (delivery manifests untouched). This is NOT flow accounting - it subtracts one witnessed term from the assume-start-loaded basis so the depot is not debited for ore the environment provided ("the hub loses what the colony gains" stays honest). Full per-provenance attribution replaces this in M3. KSC routes: `CostManifest` semantics unchanged (funds basis is OQ1's job).
- **D9 - funds basis = OQ1 recommendation,** implemented as a new overload `RouteFundsCalculator.ComputeDispatchFundsCost(vesselSnapshot, startResourceManifest, partCostLookup, resourceUnitCostLookup)`: parts from the snapshot walk, resources from the manifest when non-null (else today's per-PART resource walk). `ComputeDispatchFundsCostForRoute` threads the source recording's `RunManifest.StartTransportResources`.
- **D10 - hash policy: sparse-append the witnessed quantities, exclude resolution/diagnostic metadata.** `RouteProofHasher` appends, ONLY when present (absent emits nothing, so every pre-M2 hash is byte-identical): `runManifest.*` (start/end resources + inventory via the existing canonical helpers) and `harvestWindows[i].{startUT,endUT,openedAtStart,closedAtStop,startRes,endRes}`. Excluded (the M1 D5 precedent, `RouteProofHasher.cs:116-122`): harvest-site location fields and `ActiveConverters` strings - diagnostic/resolution metadata, not the witnessed transfer. Rationale for inclusion at all: harvested gains and run manifests now GATE eligibility and size `CostManifest`; a rewrite that drops them must flip `SourceChanged` (fail closed), exactly what the hash exists for.
- **D11 - fixture/seam strategy for mod resources.** No live `PartResourceLibrary` in xUnit, so: funds = injected lookups (already), transferability = `DefinitionLookupOverrideForTesting` (D1), capture/codec/hasher = name-agnostic already (fixtures via `VesselSnapshotBuilder` PART/RESOURCE nodes). A small `CrpFixtures` constants class (names like `Karbonite`, `MetallicOre`, `Uraninite`, `Supplies` with representative unit costs incl. one zero-cost and one undefined) keeps the CRP names consistent across suites. No Unity shim, no new harness.
- **D12 - deferred-and-fail-closed list (restated):** BG-leg harvest capture; in-window (docked) harvest admission; inventory closure; consumed/residual checks; N-window accounting. Each fails closed through existing or new named rejections - nothing silently passes.
