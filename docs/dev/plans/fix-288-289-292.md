## Bug #292 — F9 quickload drops post-merge recordings (game-breaking)

### Root cause (verified)

`quicksave.sfs` is **never refreshed by Parsek**. F5 captures the game state at time T1 with N recordings. Subsequent merges/splits add new recording IDs to the in-memory tree, but `quicksave.sfs` keeps the T1 snapshot. F9 loads the T1 snapshot, the load handler logs `removed stale committed copy of tree 'Kerbal X' (…, 17 recording(s)) — active-tree restore takes precedence`, and the in-memory 17-rec committed tree is replaced by the loaded 12-rec tree. The 5 outer-space `.prec` files remain on disk as orphans.

The "active-tree restore takes precedence" path is **correct behavior**: the loaded tree must be the source of truth after F9, otherwise rewind/quickload semantics break. The bug is that the loaded tree is stale.

### Fix (rev 2)

After the user clicks "Merge to Timeline" in the merge dialog and the post-merge optimization pass completes, refresh `quicksave.sfs` so subsequent F9 reloads see the new state. Use the existing `GamePersistence.SaveGame("quicksave", ...)` pattern from `FlightRecorder.CaptureRewindSave` (FlightRecorder.cs:3738).

**The refresh must NOT happen inside `RunOptimizationPass` itself.** `RunOptimizationPass` is also called from `ParsekScenario.OnLoad` at lines 1085 and 1197 (cold load + post-revert dispatch), and from `ChainSegmentManager`. Calling `GamePersistence.SaveGame` from inside an OnLoad callback would re-enter `OnSave` on every `ScenarioModule`, which is unsafe and could deadlock or corrupt scenario state.

**The refresh must be at the merge dialog entry point only.** The MergeDialog "Merge to Timeline" button at `MergeDialog.cs:354-374` is the only USER-initiated path that adds new recording IDs to the in-memory tree. The user expects state to advance as a result of clicking the button. Auto-pass optimizations (load-time, chain promotion) do NOT need quicksave refresh because they don't add new IDs that the player would expect to survive a quickload.

### Files touched (rev 2)

- `Source/Parsek/RecordingStore.cs` — new internal helper `RefreshQuicksaveAfterMerge(string reason, int recordingCount)` that:
  - Guards on `HighLogic.LoadedScene == GameScenes.LOADING` (no-op during startup)
  - Guards on `HighLogic.CurrentGame == null`
  - Calls `GamePersistence.SaveGame("quicksave", HighLogic.SaveFolder, SaveMode.OVERWRITE)`
  - Logs `[Quicksave] Refreshed quicksave.sfs after {reason} (tree has {N} recording IDs)` at Info
  - Wraps in try/catch and logs warn on failure (don't break the merge if the save fails)
  - Uses an `internal static System.Func<string, string, SaveMode, string> SaveGameForTesting` field defaulting to `GamePersistence.SaveGame`. Tests override it to a no-op lambda that records the call.
- `Source/Parsek/MergeDialog.cs` lines 356-374 — after `RecordingStore.RunOptimizationPass()` (line 360) and before `LedgerOrchestrator.NotifyLedgerTreeCommitted(tree)` (line 361), insert:
  ```csharp
  RecordingStore.RefreshQuicksaveAfterMerge("merge dialog Tree Merge", tree.Recordings.Count);
  ```
  This is the ONLY call site. Do NOT add the call to `RunOptimizationPass`, `CommitPendingTree`, or `OnLoad`.
- Do **not** touch `RunOptimizationPass`, `ParsekScenario.OnLoad`, or `ChainSegmentManager`.

### Why this is safe

- Refresh fires only after a USER-initiated merge (the player clicks "Tree Merge" in the dialog) — not on every save or every flight frame.
- The merge already changes the tree state in a player-visible way; refreshing the quicksave to match is the obvious "save my latest state" behavior.
- The trade-off is that the player's "F5 checkpoint" UT advances when a merge happens. This is acceptable because: (a) the player initiated the merge, (b) the alternative is silent data loss.
- If the merge produced no new IDs (split-only optimization with zero candidates), the helper does nothing.

### Tests (rev 2)

Add to new `Source/Parsek.Tests/QuicksaveRefreshTests.cs`:

- `RefreshQuicksaveAfterMerge_LoadingScene_NoOp` — set `HighLogic.LoadedScene = GameScenes.LOADING` (or seam it via test override), call helper, verify the override `SaveGameForTesting` lambda was NOT invoked.
- `RefreshQuicksaveAfterMerge_NormalCall_InvokesSaveGame_LogsAtInfo` — install a no-op `SaveGameForTesting` lambda that records the call args, invoke the helper, verify the lambda received `("quicksave", expectedSaveFolder, SaveMode.OVERWRITE)` and that the Info log line fired with the reason and recording count.
- `RefreshQuicksaveAfterMerge_SaveGameThrows_LogsWarn` — install a `SaveGameForTesting` lambda that throws, verify the helper catches the exception and logs a warn line.
- The MergeDialog call site itself is NOT directly testable (Unity dialog code), so coverage is via the in-game test for #292 (see "Manual playtest reproduction" below).

### Risk (rev 2)

- **Re-entrancy** — addressed by placing the refresh ONLY at the merge dialog button handler, never inside `RunOptimizationPass` or any OnLoad path. The MergeDialog runs AFTER OnLoad has fully returned (it's triggered by user click in spacecenter or tracking station). The scene guard adds defense-in-depth.
- **The refresh moves the player's F5 timestamp forward.** This is intentional and only happens when the user explicitly clicked "Merge to Timeline". Document in CHANGELOG so users know. Players who rely on F5 as a "manual checkpoint" should F5 again BEFORE clicking Merge if they want a pre-merge checkpoint.
- **`HighLogic.SaveFolder` may differ from the active save** in unusual cases (e.g., during a save migration). The existing `FlightRecorder.CaptureRewindSave` uses the same value, so this is acceptable.
- **Fail-safe behavior:** if `SaveGame` throws, we log a warning and continue. The player loses the quicksave-refresh benefit but the merge still completes. The next persistent.sfs save (which fires on next scene change) will capture the post-merge state, and the next manual F5 will refresh the quicksave.

### Out of scope

- Orphan `.prec` recovery for the user's currently-affected save (their 5 outer-space `.prec` files are still on disk). Recovery UI is a separate feature, deferred to a follow-up PR. The user can manually salvage by copying the 5 `.prec` files into a fresh save.

## Bug #289 — End-of-mission spawn never fires (vessel snapshot situation stale)

### Root cause (verified by log timeline)

The flow that produces the bug:

1. Active recording `a8782158` reaches splashdown at UT~49533. Vessel `vessel.situation == SPLASHED`.
2. Player initiates scene exit (rewind from flight). KSP fires `OnSceneChangeRequested`.
3. **KSP runs OnSave BEFORE Parsek's `FinalizeTreeRecordings`.** The KSP.log lines confirm this:
   - `00:53:23.498` — `[#66][OnSave:post]` — Parsek's OnSave returns. The .sfs file at this point has `a8782158.terminalState = null` because finalize hasn't run yet.
   - `00:53:23.504` — `Game State Saved to saves/s34/persistent` — KSP commits the save.
   - `00:53:23.505` — `[#67][OnSceneChangeRequested]` — Parsek begins post-save scene-exit work.
   - `00:53:23.507` — `FinalizeTreeRecordings: rec='a8782158…' terminal=Splashed snapshot=True leaf=True` — finalize NOW sets `TerminalStateValue = Splashed` IN MEMORY ONLY.
4. Scene change to SPACECENTER. In-memory tree has `a8782158.terminal = Splashed`. Disk .sfs has `a8782158.terminal = null`.
5. `00:53:27` — A `RecordingTree.Load` reads the .sfs back into memory. The in-memory `a8782158.terminal` is now null (overwritten by load).
6. `00:53:33` — Merge dialog runs. `SessionMerger.MergeTree` copies `srcRec.TerminalStateValue` to `merged.TerminalStateValue` — both null.
7. `RecordingOptimizer.SplitAtSection` runs on `a8782158`. Lines 430–435 transfer terminal state from original (first half) to second (second half) — both end up null because the source was null.
8. Spawn check on the second half (`#14 "Kerbal X"`) hits `GhostPlaybackLogic.ShouldSpawnAtRecordingEnd` line 2486:

   ```csharp
   bool terminalOverridesUnsafe = rec.TerminalStateValue == TerminalState.Landed ||
       rec.TerminalStateValue == TerminalState.Splashed ||
       rec.TerminalStateValue == TerminalState.Orbiting;
   if (!terminalOverridesUnsafe && IsSnapshotSituationUnsafe(rec.VesselSnapshot))
       return (false, "snapshot situation unsafe (FLYING/SUB_ORBITAL)");
   ```

   `terminalOverridesUnsafe == false` (terminal is null after the load round-trip), the snapshot's `sit` field is "FLYING" (captured at recording start, never updated), check fires, spawn blocked.

The override exists (added by PR #161 and PR #171 for prior incidents) but only fires when the terminal state survives the save/load round-trip. For active recordings on scene exit, it doesn't.

### Fix (rev 2 — three-part)

The re-snapshot alone is NOT sufficient. The reviewer caught two critical issues:

1. The user's flow goes through `CommitTreeSceneExit` (autoMerge OFF branch at `ParsekFlight.cs:961-970`), which preserves snapshots — good. But on the autoMerge ON branch (`CommitTreeSceneExit` function at lines 5962-5983), snapshots are explicitly nulled at line 5975. We will NOT touch that branch in this PR — autoMerge ON players hit a different design decision and won't get the fix.
2. **`TryRestoreActiveTreeNode` (`ParsekScenario.cs:1723-1782`) loads tree from disk and stashes it as pending, REPLACING the in-memory stash from `CommitTreeSceneExit`.** Lines 1755 + 1767 explicitly Pop and re-StashPending. So the in-memory re-snapshot is *thrown away* on the next OnLoad unless it's also persisted to the disk sidecar before OnLoad runs.

The fix is therefore **three parts**, not one:

#### Part A — replace `!isSceneExit` gate with `stableTerminal` + `MarkFilesDirty`

In `ParsekFlight.cs:6121-6131`, replace:

```csharp
// Re-snapshot live vessels for Commit Flight path (fresh state)
if (!isSceneExit)
{
    ConfigNode freshSnapshot = VesselSpawner.TryBackupSnapshot(vessel);
    if (freshSnapshot != null)
    {
        rec.VesselSnapshot = freshSnapshot;
        if (rec.GhostVisualSnapshot == null)
            rec.GhostVisualSnapshot = freshSnapshot.CreateCopy();
    }
}
```

with:

```csharp
// Re-snapshot live vessels whenever terminal state is stable so the snapshot's
// sit field reflects the actual splashdown/landing/orbit state. Without this,
// the snapshot's sit field is stale from recording start (FLYING/SUB_ORBITAL)
// and the spawn-at-end safety check blocks spawn even when the recording's
// terminal state is correctly Splashed. (#289)
//
// Applies to scene-exit too because OnSave fires BEFORE FinalizeTreeRecordings
// on scene exit, so the disk .sfs never sees the post-finalize state — only
// in-memory updates can carry it. The autoMerge OFF path at ParsekFlight.cs:961
// preserves snapshots (it's the path the merge dialog uses), so re-snapshotting
// during isSceneExit is safe and useful there.
bool stableTerminal = rec.TerminalStateValue == TerminalState.Landed
    || rec.TerminalStateValue == TerminalState.Splashed
    || rec.TerminalStateValue == TerminalState.Orbiting;
if (stableTerminal)
{
    ConfigNode freshSnapshot = VesselSpawner.TryBackupSnapshot(vessel);
    if (freshSnapshot != null)
    {
        rec.VesselSnapshot = freshSnapshot;
        if (rec.GhostVisualSnapshot == null)
            rec.GhostVisualSnapshot = freshSnapshot.CreateCopy();
        rec.MarkFilesDirty();  // Critical: ensure the fresh snapshot reaches the sidecar
        ParsekLog.Info("Flight",
            $"FinalizeIndividualRecording: re-snapshotted '{rec.RecordingId}' " +
            $"with stable terminal state {rec.TerminalStateValue} " +
            $"(vessel.situation={vessel.situation}, isSceneExit={isSceneExit})");
    }
}
```

The `MarkFilesDirty()` call is the reviewer's MUST FIX item 2. Without it, the next `SaveRecordingFiles` invocation skips this recording (it iterates dirty recordings only) and the fresh snapshot never reaches `<id>_vessel.craft`.

#### Part B — force-write dirty sidecars after finalize on scene exit (autoMerge OFF only)

In `ParsekFlight.cs:961-970` (the `else if (!ParsekScenario.IsAutoMerge)` branch), after `FinalizeTreeRecordings(activeTree, commitUT, isSceneExit: true)`, add:

```csharp
// #289: KSP's OnSave already fired before scene-exit finalize ran, so the .sfs
// captured the pre-finalize state. We need to force-write any sidecars (esp.
// fresh snapshots from the re-snapshot in Part A) to disk BEFORE the next
// OnLoad triggers TryRestoreActiveTreeNode, which would otherwise hydrate the
// stashed tree from stale sidecars.
int forcedWrites = 0;
foreach (var rec in activeTree.Recordings.Values)
{
    if (rec.FilesDirty)
    {
        if (RecordingStore.SaveRecordingFiles(rec))
            forcedWrites++;
    }
}
if (forcedWrites > 0)
    ParsekLog.Info("Flight",
        $"CommitTreeSceneExit (autoMerge off): force-wrote {forcedWrites} dirty sidecar(s) " +
        $"after finalize so post-finalize state survives the next OnLoad");
```

This bypasses the OnSave timing issue. The disk sidecar now has the fresh snapshot. When `TryRestoreActiveTreeNode` runs on the next OnLoad and calls `LoadRecordingFiles(rec)` (line 1741), it reads the fresh snapshot back into memory.

#### Part C — preserve snapshot on disk for the `TryRestoreActiveTreeNode` round-trip

`TryRestoreActiveTreeNode` calls `RecordingStore.LoadRecordingFiles(rec)` for each recording in the loaded tree. `LoadRecordingFiles` reads the `<id>_vessel.craft` sidecar into `rec.VesselSnapshot`. Verify (read-only step in implementation) that `LoadRecordingFiles` does not skip the snapshot file when the recording's metadata says terminal state is null. If it does skip, fix it to always load the sidecar.

### Why this fix works for the user's path

Trace through the user's playtest with the three-part fix in place:

1. `00:53:23` — Scene exit FLIGHT → SPACECENTER. KSP's OnSave fires (writes pre-finalize state to .sfs).
2. `00:53:23.505` — `CommitTreeSceneExit (autoMerge off)` runs.
3. `00:53:23.507` — `FinalizeTreeRecordings` sets `a8782158.terminal = Splashed`. **Part A** re-snapshots, sets `MarkFilesDirty()`.
4. **Part B** force-writes the dirty sidecar — the `_vessel.craft` for `a8782158` now has the fresh snapshot with `sit=SPLASHED` on disk.
5. `00:53:23.507` — `StashPendingTree(activeTree)` — in-memory stash with fresh snapshot.
6. Scene reload. `00:53:27` — OnLoad in spacecenter.
7. `TryRestoreActiveTreeNode` loads tree from .sfs (terminal state still null, that's fine), then **Part C** ensures `LoadRecordingFiles` reads the fresh sidecar, restoring `rec.VesselSnapshot.sit = "SPLASHED"`.
8. `00:53:30` — Merge dialog opens with the in-memory tree, which now has the fresh snapshot.
9. `00:53:33` — `SessionMerger.MergeTree` runs. `ApplySessionMergeToRecordings` overwrites only `TrackSections/PartEvents/SegmentEvents/Points/OrbitSegments` — does NOT touch `VesselSnapshot`. Original `a8782158.VesselSnapshot` survives merged with fresh `sit=SPLASHED`.
10. `RecordingOptimizer.SplitAtSection` runs. Line 431 transfers `original.VesselSnapshot` (fresh, `sit=SPLASHED`) to `second.VesselSnapshot`. Line 432 nulls original.
11. Spawn check on second half: `IsSnapshotSituationUnsafe(snapshot)` reads `sit=SPLASHED` → returns `false` → spawn ALLOWED.

Bug fixed. The terminal state being null in memory no longer matters — the snapshot's `sit` field carries the truth.

## Bug #287 — Map icon hidden until W (Watch) is pressed

### Root cause (verified)

`Recording.TerminalOrbit*` fields (8 individual fields, decomposed Keplerian elements) are only populated by `ParsekFlight.PopulateTerminalOrbitFromLastSegment` at `ParsekFlight.cs:6510-6520`, which is **only called from finalize on the "vessel not found → marked Destroyed" path** (lines 6144, 6178). For non-terminal in-progress recordings, the cache is empty even when `OrbitSegments` has valid orbital data.

`GhostMapPresence.HasOrbitData(Recording)` at `GhostMapPresence.cs:87-107` checks the cache:

```csharp
bool hasOrbit = !string.IsNullOrEmpty(rec.TerminalOrbitBody)
    && rec.TerminalOrbitSemiMajorAxis > 0;
```

Returns `false` for empty cache → no map vessel created → no map icon. Pressing W triggers `WatchModeController` which reads `Recording.HasOrbitSegments` directly (`WatchModeController.cs:160`), bypassing the broken cache.

### Fix (rev 2 — simpler, no interface change)

The reviewer's SHOULD FIX 1 caught a bug in the original plan: `RecordingTree.LoadRecordingFrom` runs BEFORE the sidecar `.prec` load. At `LoadRecordingFrom` time, `OrbitSegments` is still empty (it gets populated later by `LoadRecordingFiles`). So calling `PopulateTerminalOrbitFromLastSegment` from `LoadRecordingFrom` is a no-op. The reviewer's SHOULD FIX 3 also pointed out that adding `LastOrbitSegment` to `IPlaybackTrajectory` is unnecessary because the interface already exposes `OrbitSegments` and `HasOrbitSegments`.

Revised two-part fix:

1. **Eager populate after sidecar load** — call `PopulateTerminalOrbitFromLastSegment(rec)` at the end of `RecordingStore.LoadRecordingFiles`, AFTER `DeserializeTrajectoryFrom` has populated `rec.OrbitSegments`. This is the only load path that matters because `OrbitSegments` lives in the `.prec` sidecar in v3 format. The existing `PopulateTerminalOrbitFromLastSegment` helper has a "do not overwrite if already set" guard, so it's safe to call unconditionally.
2. **Read-side fallback in both `HasOrbitData` overloads** — when `TerminalOrbitBody` is null/empty, fall back to reading the last `OrbitSegment` directly. The interface version uses `traj.OrbitSegments[traj.OrbitSegments.Count-1]` (no new interface property needed). Defensive: handles in-memory recordings that grow segments after load.

### Files touched (rev 2)

- `Source/Parsek/RecordingStore.cs` — at the end of `LoadRecordingFiles` after `DeserializeTrajectoryFrom` returns (around line 3170 per reviewer's notes), call `ParsekFlight.PopulateTerminalOrbitFromLastSegment(rec)` if `string.IsNullOrEmpty(rec.TerminalOrbitBody) && rec.OrbitSegments != null && rec.OrbitSegments.Count > 0`. Log `[Parsek][Verbose][RecordingStore] Eager-populated TerminalOrbit for {recId} from last orbit segment (body={body}, sma={sma})` so we can verify in playtest logs.
- `Source/Parsek/GhostMapPresence.cs:87-107` — `HasOrbitData(Recording)`: when `TerminalOrbitBody` is null/empty, check `rec.OrbitSegments != null && rec.OrbitSegments.Count > 0`. If yes, read the last segment and use its `bodyName` and `semiMajorAxis`. Log `[Parsek][Verbose][GhostMap] HasOrbitData(Recording): cache empty, fallback from OrbitSegments[last] body={body} sma={sma} result={true/false}`.
- `Source/Parsek/GhostMapPresence.cs:113-132` — `HasOrbitData(IPlaybackTrajectory)`: same fallback using `traj.OrbitSegments` and `traj.HasOrbitSegments`. The interface already exposes both — verified by reading `IPlaybackTrajectory.cs`.

- **Do NOT** add `LastOrbitSegment` to `IPlaybackTrajectory` — unnecessary since the interface already exposes `OrbitSegments`.
