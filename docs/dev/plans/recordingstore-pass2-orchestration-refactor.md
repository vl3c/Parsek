# RecordingStore Pass 2 Orchestration Refactor Plan

Status: first group-store slice implemented in this branch. Later slices remain proposal only.

## Goal

Plan and track a zero-logic extraction for the remaining `RecordingStore` orchestration after Pass 2 storage/codecs were split out.

Hard boundaries:

- Do not change recording IDs.
- Do not change file paths.
- Do not change sidecar format.
- Do not change save/load behavior.
- Do not change deletion semantics.
- Do not change rewind behavior.
- Do not change UI-facing contracts.

These owners already cover storage and codec work and should not be re-expanded in `RecordingStore`:

- `SidecarFileCommitBatch`

## Remaining Ownership Map

`RecordingStore` still owns these orchestration clusters:

- Timeline state hub: committed recordings, committed trees, pending tree slot, state version, commit/add/remove/clear helpers.
- Tree commit pipeline: session merge, rewind provisional promotion, staging rewind point promotion, group assignment/adoption, superseded terminal spawn marking, final commit flush and milestone creation.
- Chain helpers: mid-segment/end lookup, chain recording lookup/removal/validation, loop state, segment phase labels.
- Group orchestration: auto-generated tree groups, standalone auto-assignment tracking, UI group mutation APIs.
- Optimizer orchestration: merge/split/trim execution, tree dictionary and branch-point rewrites, loop-sync parent indexing, dirty-file flush.
- Deletion and cleanup: recording sidecar deletion, individual/full recording deletion, known recording ID collection, orphan sidecar cleanup.
- Rewind orchestration: playback reset, applied markers, rewind owner lookup, can-rewind/can-fast-forward gates, rewind save preprocessing, rewind initiation.
- Compatibility wrappers: trajectory, manifest, and sidecar methods that already delegate to extracted owners.

## Required Review Fixes

The GPT-5.5 xhigh review found three required adjustments to the first-slice plan:

1. Add `RecordingGroupStore.ResetForTesting()` and call it from `RecordingStore.ResetForTesting()`.

   `autoAssignedStandaloneGroupsByRecordingId` currently gets cleared by `RecordingStore.ResetForTesting()`. After moving that dictionary, stale auto-assigned group markers must not leak between tests or affect orphan adoption through `TryGetAutoAssignedStandaloneGroup` / `ShouldAdoptRecordingIntoTreeGroup`.

   `RecordingGroupStore.ResetForTesting()` should be silent. Current `RecordingStore.ResetForTesting()` does not log, so adding a reset log would be a behavior change.

2. Preserve the existing log subsystem tag.

   Moved methods should keep logging with:

   ```csharp
   ParsekLog.Info("RecordingStore", ...)
   ParsekLog.Warn("RecordingStore", ...)
   ParsekLog.Verbose("RecordingStore", ...)
   ```

   Do not switch these logs to `"RecordingGroupStore"` in the zero-logic slice. Tests and log-contract expectations currently observe the `"RecordingStore"` tag.

3. Preserve tree commit ordering.

   `CommitTree` must keep group assignment before orphan adoption:

   ```csharp
   AutoGroupTreeRecordings(tree);
   AdoptOrphanedRecordingsIntoTreeGroup(tree);
   ```

4. Preserve the auto-assigned standalone group invariant.

   `Recording.AutoAssignedStandaloneGroupName` is the persisted authority, written during ConfigNode load. `autoAssignedStandaloneGroupsByRecordingId` is only the in-memory mirror for recordings built directly in tests or runtime paths without a persisted field. `TryGetAutoAssignedStandaloneGroup` must remain field-first and keep reconciling the mirror from the field.

5. Keep `LegacyGloopsGroupName` on `RecordingStore`.

   `LegacyGloopsGroupName` is load-migration metadata and is not used by the group methods in this slice. `GloopsGroupName` also remains on `RecordingStore` for compatibility.

6. Do not introduce `BumpStateVersion()` calls.

   Current group mutation methods do not bump `StateVersion`. Zero-logic extraction must preserve that, even if a future reviewer is tempted to treat group membership as committed-state mutation.

## Caller and Wrapper Strategy

First slice:

- Keep every existing `RecordingStore` public/internal method signature.
- Keep private `RecordingStore` wrappers where needed for the tree commit pipeline.
- Delegate implementation to `RecordingGroupStore`.
- Pass `committedRecordings` and `committedTrees` explicitly.
- Keep `RecordingStore.GloopsGroupName` as the public/internal constant source for this slice.
- Keep `RecordingStore.LegacyGloopsGroupName` in `RecordingStore`; it is not part of this slice.
- Do not update UI code or tests to call `RecordingGroupStore` directly.

Examples:

```csharp
public static List<string> GetGroupNames()
    => RecordingGroupStore.GetGroupNames(committedRecordings);

internal static string GenerateUniqueGroupName(string baseName)
    => RecordingGroupStore.GenerateUniqueGroupName(baseName, committedRecordings);

private static void AutoGroupTreeRecordings(RecordingTree tree)
    => RecordingGroupStore.AutoGroupTreeRecordings(tree, committedRecordings);
```

For methods that mutate auto-assignment state, keep the `RecordingStore` wrappers:

```csharp
internal static void MarkAutoAssignedStandaloneGroup(Recording rec, string groupName)
    => RecordingGroupStore.MarkAutoAssignedStandaloneGroup(rec, groupName);

internal static void MarkAutoAssignedStandaloneGroupForTesting(Recording rec, string groupName)
    => RecordingGroupStore.MarkAutoAssignedStandaloneGroup(rec, groupName);
```

`RecordingStore.ResetForTesting()` must include:

```csharp
RecordingGroupStore.ResetForTesting();
```

## Rollback Plan

The first slice should be wrapper-only with no caller migration. Rollback is:

1. Delete `RecordingGroupStore.cs`.
2. Restore moved methods and `autoAssignedStandaloneGroupsByRecordingId` into `RecordingStore.cs`.
3. Remove the `RecordingGroupStore.ResetForTesting()` call from `RecordingStore.ResetForTesting()`.

No save migration, sidecar cleanup, recording ID repair, or UI migration should be required because the slice must not alter serialized data, file paths, deletion behavior, rewind behavior, or public/internal `RecordingStore` contracts.
